# 模糊测试与差异测试

cuda-oxide 的正常路径通过示例测试很容易：`vecadd`、`gemm`、`sharedmem` 等证明了已知程序仍然有效。但编译器不仅仅会在已知程序上失败。它们会在没有人想到手动编写的奇怪角落失败 —— 分支之后的类型转换、调用终止符内部的元组、没人邀请的整数宽度。

这就是模糊测试发挥作用的地方。

cuda-oxide 使用一个基于 rustlantis 的小型工具来生成随机的自定义 MIR 程序，通过普通的 Rust CPU 后端和 cuda-oxide 的 GPU 后端运行它们，并比较中间值的紧凑跟踪。目标不是证明 Rust 在 GPU 上的全部正确性。目标更谦逊，也更有用：用我们未手动编写的程序来压力测试 MIR 导入器、降级流水线、LLVM 导出、PTX 生成和运行时执行。

---

## 我们在比较什么

乍一看，“比较 CPU 和 GPU 执行”听起来很可疑。CPU 和 GPU 有不同的执行模型：一个标量线程与数千个 SIMT 通道、发散的控制流、不同的内存空间、不同的同步规则。一个通用的 CPU 与 GPU 语义比较将是一个非常花哨的自欺欺人方式。

因此，该工具刻意避免了这一点。

它将生成的 MIR 作为标量 GPU 程序运行：

```text
<<<1, 1>>>
```

一个块。一个线程。没有线程间通信。没有调度问题。GPU 被用作相同标量 MIR 的第二个代码生成目标，而不是作为并行编程模型。

比较是：

```text
相同的生成 MIR
  -> 正常的 rustc/LLVM CPU 执行
  -> cuda-oxide -> LLVM IR -> PTX -> CUDA 执行
  -> 比较跟踪哈希
```

如果哈希匹配，则生成的程序在两条路径上观察到相同的中间值序列。如果它们不同，则 cuda-oxide 路径中的某些地方值得关注。

> **注意：**
> 跟踪有意保持紧凑。不是将每个中间值复制回主机，而是每个 `dump_var(...)` 调用使用逐字节哈希将值折叠成一个 `u64`。这使得生成的测试运行成本低廉且易于比较。

## 组成部分

模糊测试设置分为四个部分：

| 部分     | 角色                 |
| :----------| :------- |
| `crates/fuzzer`    | 共享跟踪 API 和 vendored rustlantis    |
| `crates/fuzzer/tools/mir_generator.py` | 种子到 `generated_case.rs` 的适配器            |
| `crates/fuzzer/tools/run_seed.py`    | 批量运行器和产物记录器          |
| `rustlantis-smoke`     | 稳定的 CPU/GPU 执行工具        |

`crates/fuzzer` 是一个普通的工作区 crate，但其库接口是 `no_std`。设备代码从那里导入 `trace_reset`、`trace_finish` 和泛型 `dump_var`。实际的 rustlantis 源代码被 vendored 在 `crates/fuzzer/rustlantis` 下；它作为外部生成器被调用，而不是作为 Rust 库依赖项使用。

`rustlantis-smoke` 示例位于 `crates/rustc-codegen-cuda/examples/` 下。它拥有主机/GPU 启动逻辑和一个小的手写健全性测试，然后包含一个生成的文件：

```text
crates/rustc-codegen-cuda/examples/rustlantis-smoke/src/generated_case.rs
```

模糊测试工具为每个种子重写该文件。示例中的其他所有内容保持稳定。这使得工具易于审查：如果一个种子失败，生成的 MIR 被隔离在一个地方。

## 种子流水线

一个种子遵循以下路径：

1. **生成。** rustlantis 接收一个数字种子和一个小的配置。相同的种子加上相同的配置意味着相同的自定义 MIR 程序。
2. **提取。** `mir_generator.py` 提取第一个生成的 `#[custom_mir]` 函数。
3. **适配。** rustlantis 发出类似 `dump_var(a, b, c)` 的调用。适配器将它们重写为元组局部变量和一个泛型的 `fuzzer::dump_var(...)` 调用，因为自定义 MIR 调用操作数对元组表达式很挑剔。
4. **注入。** 适配后的函数和一个小的包装器被写入 `src/generated_case.rs`。
5. **运行。** `cargo oxide run rustlantis-smoke` 执行 CPU 预言机和 GPU kernel。
6. **分类。** `run_seed.py` 记录种子是通过、不匹配、编译失败还是超出适配器当前的支持范围。

当前检入的生成用例使用种子 `19`，因为它展示了我们想要从工具中获得的重要属性：多个中间值转储，而不仅仅是一个最终值。

```rust
__rl_dump0 = (Move(_1), Move(_2), Move(_3), Move(_4));
Call(_9 = dump_var(Move(__rl_dump0)), ReturnTo(bb4), UnwindUnreachable())

__rl_dump1 = (Move(_6),);
Call(_9 = dump_var(Move(__rl_dump1)), ReturnTo(bb5), UnwindUnreachable())
```

这意味着最终的跟踪哈希包括程序中某个点的几个值和后面另一个点的另一个值。它仍然是紧凑的，但它不再仅仅是“返回值匹配吗？”

## 运行它

运行一个种子：

```bash
python3 crates/fuzzer/tools/run_seed.py --seed 192
```

运行一批：

```bash
python3 crates/fuzzer/tools/run_seed.py --start 0 --count 20 --keep-going
```

有用的标志：

- `--keep-going`：在失败种子后继续。
- `--keep-logs`：也为通过的种子写入日志。
- `--no-build`：重用已构建的 rustlantis 生成器。
- `--append-summary`：追加到现有摘要而不是替换它。

默认情况下，`summary.jsonl` 在每次运行开始时被替换。这使得它可以回答一个明显的问题：“我刚刚进行的运行发生了什么？”如果你想要历史记录，选择 `--append-summary`。

## 阅读结果

运行器为每个种子打印一行，然后是一个完整的摘要：

```text
results:
  seed 0: UNSUPPORTED [adapter] unsupported dumped type for Stage 2 adapter: u128 (...)
  seed 1: COMPILE_FAIL [backend] Unsupported construct: Type translation not yet implemented for: RigidTy(Char) (...)
summary: COMPILE_FAIL=1, UNSUPPORTED=1
```

状态含义：

| 状态    | 含义          |
| :--------- | :----------|
| `PASS` | CPU 和 GPU 跟踪匹配              |
| `MISMATCH` | CPU 和 GPU 跟踪不同       |
| `COMPILE_FAIL [backend]` | 适配器产生了一个用例，但 cuda-oxide 失败了    |
| `UNSUPPORTED [adapter]`  | rustlantis 生成了 MIR，但适配器拒绝了它      |

`MISMATCH` 是最需要认真对待的结果。两条路径都编译并运行了，但观察到了不同的值。这是一个潜在的后端正确性错误。

`COMPILE_FAIL [backend]` 意味着生成的用例通过了适配器并进入了 cuda-oxide。失败可能仍然是预期的 —— 例如，当前不支持的 MIR 类型 —— 但后端是拒绝它的组件。

`UNSUPPORTED [adapter]` 意味着 rustlantis 生成了一个程序，但我们的适配器拒绝将其变成一个烟雾测试用例。例如：

```text
unsupported dumped type for Stage 2 adapter: u128
```

这通常意味着生成的 MIR 有一个 `dump_var(...)` 包含我们的跟踪 API 还不知道如何哈希的类型。目前跟踪支持：

```text
bool, i8, i16, i32, i64, u8, u16, u32, u64
```

它尚不支持 `u128`、`i128`、`usize`、`isize` 或 `char`。因此，许多适配器级别的“不支持”情况不是“坏的 MIR”，也不是 cuda-oxide 的错误。它们只是模糊测试工具尚未成熟的领域。编译器和人一样，在处理 `u128` 之前需要吃点零食。

## 产物

每个种子的日志位于：

```text
crates/fuzzer/artifacts/
```

失败的日志包括：

- 种子
- 状态和阶段
- 原因
- 返回码
- 命令
- 完整的命令输出
- 生成的 `generated_case.rs` 快照（如果存在）

生成的快照很重要。如果后端失败出现在 CI 或长时间的批量运行中，日志足以看到触发它的确切 MIR 形状的程序。种子可以让你重新生成它，但快照为你节省了一次往返。

---

## 当前限制

当前的配置故意很小。它使第一阶段专注于标量自定义 MIR 和后端基础设施，而不是同时处理所有 Rust 结构。这就是为什么许多早期种子被分类为 `UNSUPPORTED [adapter]`。

扩展计划是渐进式的：

1. 为更多标量类型添加跟踪支持（`u128`、`i128`、`usize`、`isize`）。
2. 决定是否以及如何在 cuda-oxide 的类型翻译中支持 `char`。
3. 扩展控制流和转换覆盖率。
4. 添加数组、元组，最终添加结构体/枚举。
5. 为失败的种子添加最小化。

这个顺序是经过深思熟虑的。一个第一天就生成所有东西的模糊测试器大多会产生噪音。一个一次只扩展一个轴的模糊测试器会告诉你什么坏了，以及为什么坏。友好得多。
