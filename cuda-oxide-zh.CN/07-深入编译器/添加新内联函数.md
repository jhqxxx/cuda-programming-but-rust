# 添加新内联函数

所以你想教 cuda-oxide 一个新的 GPU 技巧。也许 NVIDIA 刚发布了新指令，或者你需要一个尚未有人接线的现有 PTX 操作。好消息是：这个过程是机械式的。五个 crate，五个步骤，一旦你做过一次，大约三十分钟。

本章使用两个真实示例 —— 一个极其简单，一个有一些曲折 —— 来贯穿整个流水线，这样你可以确切地看到每个阶段发生了什么。

---

## 五阶段流水线

每个 GPU 内联函数都遵循相同的路径通过编译器：

```text
cuda-device          用户编写：thread::threadIdx_x()
    │
    ▼
mir-importer        编译器看到调用，发出一个 `dialect-nvvm` 操作
    │
    ▼
dialect-nvvm        操作作为已验证的 IR 节点存在于这里
    │
    ▼
mir-lower           将 `dialect-nvvm` 操作转换为 `dialect-llvm` 操作
    │
    ▼
dialect-llvm        导出文本 LLVM IR  →  llc 将其转换为 PTX
```

在 `mir-lower` 阶段，你选择两种策略之一：

| 策略   | 使用场景   | 示例        |
| :------- | :-------- | :--------------- |
| LLVM 内联函数调用   | LLVM 已经有内置支持    | `threadIdx_x`，线程束洗牌，屏障     |
| 内联 PTX 汇编    | 没有 LLVM 内联函数，或者你需要对 PTX 指令进行精确控制   | `trap`，`wgmma`，`tcgen05`，`mbarrier` |

下面将演示这两种策略。

---

## 示例 1：`threadIdx_x`（简单情况）

`threadIdx_x()` 是 GPU 内联函数的“Hello, World”：零个参数，一个 `u32` 结果，直接映射到单个 LLVM NVVM 内联函数。如果你能理解这个例子，你就可以添加任何简单的内联函数。

### 阶段 1 —— 在 `cuda-device` 中声明

**文件：** `crates/cuda-device/src/thread.rs`

```rust
#[inline(never)]
pub fn threadIdx_x() -> u32 {
    unreachable!("threadIdx_x called outside CUDA kernel context")
}
```

在你理解其技巧之前，这两个规则可能看起来很奇怪：

- **`#[inline(never)]`** 使函数在 MIR 中保持为一个独特的调用可见。如果 rustc 内联了它，编译器将看到 `unreachable!()` 而不是它可以拦截的调用。那将……不太有帮助。

- 主体是 `unreachable!()`，因为这个函数实际上从未运行过。编译器在任何 GPU 代码执行之前，将整个调用替换为一个 `dialect-nvvm` 操作。把这个函数看作一个占位符 —— 一个“亲爱的编译器，请在此处插入 GPU 指令”的便条。

> **提示：**
> 在函数上方的注释中记录该内联函数映射到的 LLVM IR 或 PTX。未来的你（或未来的贡献者）会感谢现在的你。

### 阶段 2 —— 定义 `dialect-nvvm` 操作

**文件：** `crates/dialect-nvvm/src/ops/thread.rs`

```rust
#[pliron_op(name = "nvvm.read_ptx_sreg_tid_x", dialect = "nvvm", format)]
pub struct ReadPtxSregTidXOp;

impl Verify for ReadPtxSregTidXOp {
    fn verify(&self, ctx: &Context) -> Result<(), Error> {
        let op = &*self.get_operation().deref(ctx);
        if op.get_num_operands() != 0 {
            return verify_err!(op.loc(), "expected 0 operands");
        }
        if op.get_num_results() != 1 {
            return verify_err!(op.loc(), "expected 1 result");
        }
        Ok(())
    }
}
```

然后注册它，以便 pliron 知道该操作存在：

```rust
pub(super) fn register(ctx: &mut Context) {
    ReadPtxSregTidXOp::register(ctx, ReadPtxSregTidXOp::parser_fn);
}
```

`Verify` trait 尽早捕获结构错误。如果某处意外地用两个操作数创建了此操作，验证会以清晰的消息失败，而不是在三个阶段之后产生垃圾 PTX。

### 阶段 3 —— 在 `mir-importer` 中识别

**文件：** `crates/mir-importer/src/translator/terminator/mod.rs`

当翻译器处理 MIR 时，每个函数调用都会经过 `try_dispatch_intrinsic()`。被调用者的完全限定域名（FQDN）通过 `extract_func_info()` 从 `CrateDef::name()` 获得，产生像 `cuda_device::thread::threadIdx_x` 这样的路径。匹配检查此 FQDN：

```rust
match name {
    "cuda_device::threadIdx_x" | "cuda_device::thread::threadIdx_x" => {
        Ok(Some(helpers::emit_nvvm_intrinsic(
            ctx,
            ReadPtxSregTidXOp::get_concrete_op_info(),
            destination, target, block_ptr, prev_op,
            value_map, block_map, loc,
        )?))
    }
    // ... 数百个其他内联函数 ...
}
```

我们同时匹配重新导出的名称（`cuda_device::threadIdx_x`）和完整的模块路径（`cuda_device::thread::threadIdx_x`），以便无论用户如何导入函数，两者都能工作。FQDN 按原样用于匹配 —— 在内联函数检查之前不会发生 `::` 到 `__` 的转换。

`emit_nvvm_intrinsic()` 辅助函数是一个泛型函数，适用于**任何**零参数、单结果的 NVVM 内联函数。它创建操作，将结果存储在值映射中，并发出到下一个基本块的分支。对于简单的内联函数，你永远不需要编写自定义发射器。

### 阶段 4 —— 降级到 `dialect-llvm`

**文件：** `crates/mir-lower/src/convert/intrinsics/basic.rs`

该操作实现了 `MirToLlvmConversion` 操作接口。在 `convert/interface_impls.rs` 中，impl 分派到一个转换器函数：

```rust
impl MirToLlvmConversion for ReadPtxSregTidXOp {
    fn rewrite(ctx, rewriter, op, operands_info) -> Result<()> {
        basic::convert_read_tid_x(ctx, rewriter, op, operands_info)
    }
}
```

然后 `basic::convert_read_tid_x` 发出 LLVM 内联函数调用：

```rust
pub fn convert_read_tid_x(
    ctx: &mut Context,
    rewriter: &mut DialectConversionRewriter,
    op: Ptr<Operation>,
    _operands_info: &OperandsInfo,
) -> Result<()> {
    let intrinsic_name = "llvm_nvvm_read_ptx_sreg_tid_x";
    let i32_ty = IntegerType::get(ctx, 32, Signedness::Signless);
    let func_ty = FuncType::get(ctx, i32_ty.into(), vec![], false);

    let call_op = call_intrinsic(ctx, rewriter, intrinsic_name, func_ty, vec![])?;
    rewriter.replace_operation_with_values(ctx, op, vec![result]);
    Ok(())
}
```

> **注意：**
> LLVM 内联函数名称使用点（`llvm.nvvm.read.ptx.sreg.tid.x`），但 pliron 标识符不能包含点。我们在内部使用下划线（`llvm_nvvm_read_ptx_sreg_tid_x`）。导出阶段会将它们转换回来。

### 阶段 5 —— 导出（无需更改）

**文件：** `crates/dialect-llvm/src/export.rs`

`CallOp` 导出器已经处理了下划线到点的转换：

```rust
let fixed_name = if name.starts_with("llvm_nvvm") {
    name.replace('_', ".")
} else {
    strip_device_prefix(&name)
};
```

由于 `threadIdx_x` 不是 convergent（它是每线程寄存器读取，不是集体操作），因此无需添加到 `is_convergent_intrinsic()`。

**最终输出：**

```llvm
declare i32 @llvm.nvvm.read.ptx.sreg.tid.x()
%v5 = call i32 @llvm.nvvm.read.ptx.sreg.tid.x()
```

经过 `llc` 后：

```text
mov.u32  %r1, %tid.x;
```

五个阶段，一条 PTX 指令。还不错。

## 示例 2：`shuffle_xor`（复杂情况）

线程束洗牌更有趣。用户传递两个参数，但底层的 LLVM 内联函数期望*四个*。编译器静默地填充额外的参数。该操作也是 **convergent**，意味着 LLVM 不得将其移过、复制或推测跨越控制流。

### 阶段 1 —— 在 `cuda-device` 中声明

**文件：** `crates/cuda-device/src/warp.rs`

```rust
#[inline(never)]
pub fn shuffle_xor(var: u32, lane_mask: u32) -> u32 {
    let _ = (var, lane_mask);
    unreachable!("shuffle_xor called outside CUDA kernel context")
}
```

和之前的模式一样。`let _ = (var, lane_mask);` 抑制未使用变量警告 —— 一个零成本的小善意。

### 阶段 2 —— 定义 `dialect-nvvm` 操作

**文件：** `crates/dialect-nvvm/src/ops/warp.rs`

```rust
#[pliron_op(name = "nvvm.shfl_sync_bfly_i32", dialect = "nvvm", format)]
pub struct ShflSyncBflyI32Op;

impl Verify for ShflSyncBflyI32Op {
    fn verify(&self, ctx: &Context) -> Result<(), Error> {
        let op = &*self.get_operation().deref(ctx);
        if op.get_num_operands() != 2 {
            return verify_err!(op.loc(), "expected 2 operands");
        }
        if op.get_num_results() != 1 {
            return verify_err!(op.loc(), "expected 1 result");
        }
        Ok(())
    }
}
```

这次是两个操作数（值和通道掩码），一个结果。

### 阶段 3 —— 在 `mir-importer` 中识别

**文件：** `crates/mir-importer/src/translator/terminator/mod.rs`

与 `threadIdx_x` 不同，这调用一个**专用发射器**，因为它需要处理用户的两个参数：

```rust
"cuda_device::warp::shuffle_xor" => Ok(Some(
    intrinsics::warp::emit_warp_shuffle_i32(
        ctx, body,
        ShflSyncBflyI32Op::get_concrete_op_info(),
        args, destination, target, ...
    )?
)),
```

**文件：** `crates/mir-importer/src/translator/terminator/intrinsics/warp.rs`

```rust
pub fn emit_warp_shuffle_i32(ctx, body, shuffle_opid, args, ...) {
    if args.len() != 2 { return error; }

    let (val, _) = translate_operand(ctx, body, &args[0], ...);
    let (lane_or_mask, _) = translate_operand(ctx, body, &args[1], ...);

    let shuffle_op = Operation::new(ctx, shuffle_opid,
        vec![u32_type.to_ptr()],          // results: [u32]
        vec![val, lane_or_mask],          // operands: [value, mask]
        vec![], 0,
    );

}
```

发射器将用户的 MIR 操作数翻译成 pliron 值 —— 通常是从每个操作数的 alloca 槽位 `mir.load` 的结果，或者对于字面参数是 `mir.constant` —— 并将它们连接到 NVVM 操作中。（这些还不是 SSA 值；`pliron::opts::mem2reg` 将在翻译完成后折叠加载/存储链。）对于具有不同参数模式的内联函数，你可以编写类似的自定义发射器。

### 阶段 4 —— 降级到 `dialect-llvm`

**文件：** `crates/mir-lower/src/convert/intrinsics/warp.rs`

这里变得有趣了。蝶形洗牌的 LLVM 内联函数接受**四个**参数，而不是两个：

```text
用户 API：     shuffle_xor(value, lane_mask)            →  2 个参数
LLVM 内联函数： shfl.sync.bfly.i32(mask, value, lane_mask, clamp) →  4 个参数
                                    ^^^^                   ^^^^^
                                    始终 -1                始终 31
```

转换器添加编译器提供的常量：

```rust
fn convert_shuffle_i32(op, ctx, intrinsic_name, clamp: i32) {
    let operands = get_operands(op, ctx)?;
    let (val, lane_or_delta) = (operands[0], operands[1]);

    let mask_val  = create_i32_const(ctx, -1);   // 所有 32 个通道参与
    let clamp_val = create_i32_const(ctx, clamp); // 完整线程束宽度

    let func_ty = FuncType::get(ctx.ctx, i32_ty.into(),
        vec![i32_ty, i32_ty, i32_ty, i32_ty], false);

    let call_op = call_intrinsic(ctx, intrinsic_name, func_ty,
        vec![mask_val, val, lane_or_delta, clamp_val]);

    map_result(op, call_op, ctx);
}
```

掩码值 `-1`（所有位设置）意味着线程束中的所有 32 个通道都参与。钳位值 `31` 意味着洗牌在完整线程束宽度处回绕。这些是几乎所有用例的正确默认值，向用户暴露它们只会是噪音。

### 阶段 5 —— 导出（添加 Convergent）

线程束洗牌是 **convergent** —— LLVM 不得相对于控制流重新排序它们。导出步骤检查 `is_convergent_intrinsic()`：

```rust
fn is_convergent_intrinsic(name: &str) -> bool {
    name == "llvm.nvvm.barrier0"
        || name.starts_with("llvm.nvvm.shfl")      // 洗牌
        || name.starts_with("llvm.nvvm.vote")       // 投票
        || name.starts_with("llvm.nvvm.mbarrier")   // 异步屏障
        || name.starts_with("llvm.nvvm.cp.async.bulk")
        // ...
}
```

如果你的新内联函数是 convergent，请在此处添加。如果你忘记了，LLVM 可能会将其提升出 `if` 块，并且你的线程束级代码将产生错误结果或死锁。这不妙。

**最终输出：**

```llvm
declare i32 @llvm.nvvm.shfl.sync.bfly.i32(i32, i32, i32, i32)

%v8 = call i32 @llvm.nvvm.shfl.sync.bfly.i32(
    i32 -1, i32 %v3, i32 %v4, i32 31) #0

attributes #0 = { convergent }
```

经过 `llc` 后：

```text
shfl.sync.bfly.b32  %r3, %r1, %r2, 31;
```

## 内联 PTX 路径

有些操作没有 LLVM 内联函数。对于这些，我们直接发出内联 PTX 汇编。辅助函数 `inline_asm_convergent()` 处理样板代码：

```rust
// wgmma fence：无输入，无输出，只有副作用
inline_asm_convergent(
    ctx, void_ty.into(), vec![],
    "wgmma.fence.sync.aligned;", ""
);

// mbarrier arrive：一个输入（指针），一个输出（令牌）
inline_asm_convergent(
    ctx, i64_ty.into(), vec![ptr_val],
    "mbarrier.arrive.shared.b64 $0, [$1];", "=l,r"
);
```

约束字符串遵循 LLVM 内联汇编语法：

| 约束 | 含义                                 |
| :--- | :----------------------------- |
| `=l` | 输出：64 位寄存器                    |
| `=r` | 输出：32 位寄存器                    |
| `r`  | 输入：32 位寄存器                    |
| `l`  | 输入：64 位寄存器                    |
| (空) | 无输入或输出（仅副作用）             |

内联汇编上的 `sideeffect convergent` 标记告诉 LLVM 别管它 —— 不要移动它，不要删除它，不要复制它。

## 端到端：完整旅程

以下是 `threadIdx_x` 经过的每一层表示，从上到下：

```text
Rust:         thread::threadIdx_x()
MIR:          _3 = threadIdx_x() -> bb1
Pliron MIR:   %v = nvvm.read_ptx_sreg_tid_x : i32
Pliron LLVM:  %v = call i32 @llvm_nvvm_read_ptx_sreg_tid_x()
LLVM IR:      %v5 = call i32 @llvm.nvvm.read.ptx.sreg.tid.x()
PTX:          mov.u32 %r1, %tid.x;
```

以及 `shuffle_xor`：

```text
Rust:         warp::shuffle_xor(val, mask)
MIR:          _5 = shuffle_xor(_3, _4) -> bb2
Pliron MIR:   %v = nvvm.shfl_sync_bfly_i32 %val, %mask : i32
Pliron LLVM:  %v = call i32 @llvm_nvvm_shfl_sync_bfly_i32(i32 -1, %val, %mask, i32 31)
LLVM IR:      %v8 = call i32 @llvm.nvvm.shfl.sync.bfly.i32(i32 -1, %v3, %v4, i32 31) #0
PTX:          shfl.sync.bfly.b32 %r3, %r1, %r2, 31;
```

六个表示。一个 Rust 函数调用变成一条 PTX 指令。中间步骤的存在是为了使每个转换都是小的、可验证的和可独立测试的。

## 快速参考清单

你需要接触的每个文件，按顺序：

1. **`cuda-device/src/<module>.rs`** —— 带有 `#[inline(never)]` 和 `unreachable!()` 主体的 `pub fn`。这是面向用户的 API。

2. **`dialect-nvvm/src/ops/<module>.rs`** —— `#[pliron_op(name = "nvvm.<name>", ...)]` 结构体，`Verify` 实现（检查操作数/结果计数），`register()` 调用。

3. **`mir-importer/src/translator/terminator/mod.rs`** —— `try_dispatch_intrinsic()` 中的 `match` 分支。对于零参数内联函数使用 `helpers::emit_nvvm_intrinsic()`，对于有参数的内联函数编写自定义发射器。

4. **`mir-lower/src/convert/interface_impls.rs`** —— 新操作的 `MirToLlvmConversion` 实现，分派到转换器函数。

5. **`mir-lower/src/convert/intrinsics/<module>.rs`** —— 转换逻辑。对于 LLVM 内联函数使用 `call_intrinsic()`，对于内联 PTX 使用 `inline_asm_convergent()`。

6. **`dialect-llvm/src/export.rs`** —— *仅当 convergent 时*：添加到 `is_convergent_intrinsic()`。

## 并排比较

| 阶段        | `threadIdx_x`（简单） | `shuffle_xor`（复杂）   |
| :---------- | :------------- | :------------------- |
| cuda-device | `fn() -> u32`     | `fn(u32, u32) -> u32`   |
| dialect-nvvm | 0 个操作数，1 个结果  | 2 个操作数，1 个结果    |
| mir-importer | 泛型辅助函数          | 自定义发射器            |
| mir-lower   | `call @intrinsic()`   | `call @intrinsic(4 args)` |
| Convergent? | 否       | 是（`#0`）              |
| PTX      | `mov.u32 %r1, %tid.x` | `shfl.sync.bfly.b32 ...` |

---

这就是整个过程。五个文件，每个都有清晰而狭窄的职责。这种模式足够机械，一旦你做过一次，添加一个新的内联函数应该只需要大约三十分钟 —— 大部分时间花在阅读 PTX ISA 规范以确定你究竟想要什么指令上。
