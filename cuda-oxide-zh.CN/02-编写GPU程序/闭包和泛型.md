# 闭包与泛型 — cuda-oxide

Rust 的零成本抽象 —— 泛型、闭包和 trait 约束 —— 在 GPU 上同样可用。这是 cuda-oxide 最独特的功能之一：你可以编写一个对任何数值类型都适用的泛型kernel，或者从主机传递闭包来自定义 GPU 行为，且完全没有运行时开销。

---

## 泛型kernel

kernel可以像任何 Rust 函数一样对类型和 trait 约束泛型。编译器将每个实例化单态化为独立的 PTX 入口点：

```rust
use cuda_device::{kernel, thread, DisjointSlice};
use core::ops::Mul;

#[kernel]
pub fn scale<T: Copy + Mul<Output = T>>(
    factor: T,
    input: &[T],
    mut out: DisjointSlice<T>,
) {
    let idx = thread::index_1d();
    let i = idx.get();
    if let Some(out_elem) = out.get_mut(idx) {
        *out_elem = input[i] * factor;
    }
}
```

### PTX 命名

每个单态化产生一个独立的 PTX 入口点。非泛型kernel保留其纯函数名。泛型kernel（包括闭包泛型kernel）获得一个 `_TID_<hex32>` 后缀，其中 `<hex32>` 是 rustc 对泛型参数*元组*的稳定类型 ID 哈希，渲染为 32 位小写十六进制字符：

| 实例化 | PTX 入口点名称 |
|--------|---------------|
| `vecadd`（非泛型） | `vecadd` |
| `scale::<f32>` | `scale_TID_<hex32>` |
| `scale::<MyType>` | `scale_TID_<hex32>` |
| `map::<f32, _>`（闭包） | `map_TID_<hex32>` |

主机启动器和设备后端都向同一个 rustc 调用请求相同的哈希，因此字符串逐字节匹配。对元组进行哈希（而非每个参数独立哈希）使得无论kernel有多少泛型参数，线上名称长度固定。借用生命周期在哈希前被擦除，因此 `&'a T` 和 `&'static T` 对相同形状 `T` 产生相同的哈希。

### 启动泛型kernel

启动时，在生成的类型化方法上指定类型参数。这会强制具体实例化，并让加载器查找匹配的 PTX 入口点：

```rust
use cuda_core::LaunchConfig;

module
    .scale::<f32>(
        &stream,
        LaunchConfig::for_num_elems(N as u32),
        2.0f32,
        &input_dev,
        &mut output_dev,
    )
    .expect("启动失败");
```

生成的方法强制 `scale::<f32>` 的单态化，因此即使它从未在 CPU 上直接调用，实例化也会出现在编译后的 PTX 中。

## 主机闭包作为kernel参数

cuda-oxide 支持从主机向 GPU 传递闭包。这实现了强大的 `map` 风格模式，其中kernel的行为由函数参数化：

```rust
#[kernel]
pub fn map<F: Fn(i32) -> i32>(f: F, input: &[i32], mut out: DisjointSlice<i32>) {
    let idx = thread::index_1d();
    let i = idx.get();
    if let Some(out_elem) = out.get_mut(idx) {
        *out_elem = f(input[i]);
    }
}
```

用闭包启动：

```rust
let factor = 3i32;
module
    .map::<_>(&stream, config, move |x| x * factor, &input_dev, &mut output_dev)
    .expect("启动失败");
```

### 闭包参数如何传输

闭包作为一个值通过启动传输 —— 不是作为捕获字段的列表。启动器推送单个驱动参数（整个闭包结构体，包含所有捕获），kernel将其作为一个 byval `.param` 接收：

```
主机          factor = 3i32; cl = move |x| x * factor
              推送一个驱动参数 ─► 闭包结构体 { factor: i32 }

GPU kernel      .entry map_TID_<hex>(
                .param .align 4 .b8 f[4],    ; 一个 byval 闭包
                .param .u64 input_ptr,        ; 切片仍是 (ptr, len)
                .param .u64 input_len,
                ...
              )
```

切片保持其 `(ptr, len)` 扁平化，因为这种形状由主机启动辅助函数和 PTX 入口点布局共享。只有按值传递的聚合参数（闭包和用户按值传递的结构体）在kernel边界上作为一个 byval 到达。

无捕获的闭包是零大小类型 —— 后端完全丢弃 `.param`，主机启动器知道跳过它，因此数据包保持对齐。

### 闭包的 PTX 命名

闭包泛型kernel与其他泛型kernel一样获得相同的 `_TID_<hex32>` 后缀。闭包的匿名类型是哈希元组中的条目之一，因此两个不同的闭包字面量 —— 即使它们具有相同的 `Fn` 签名 —— 也会产生两个独立的入口点：

```rust
move |x| x * 2     // 入口点: map_TID_abc123...
move |x| x * 3     // 入口点: map_TID_def456... (不同!)
```

## Move 闭包与引用闭包

`move` 关键字决定捕获如何传输到 GPU：

### Move 闭包（推荐默认）

```rust
let factor = 3i32;
move |x| x * factor   // `factor` 被拷贝到闭包结构体中
```

- 闭包结构体按值持有捕获（`{ factor: i32 }`）。
- kernel将 `factor` 作为 byval 闭包的常规字段读取。
- 主机变量可以在启动后释放。
- 在所有系统上工作 —— 无需特殊硬件支持。

### 引用闭包（HMM）

```rust
let factor = 3i32;
|x| x * factor   // 闭包捕获 &factor
```

- 闭包结构体包含指向 `factor` 的**主机指针**（`{ factor: &i32 }`）。
- 整个闭包仍作为一个 byval 参数传输；kernel通过**硬件管理内存（HMM）**解引用该主机指针，在访问时迁移主机页面。
- 主机变量**必须保持存活**直到kernel完成。
- 需要 HMM 支持（Turing+ GPU、Linux 6.1.24+、CUDA 12.2+）。

### 何时使用哪种

| 场景             | 使用方式                     |
| -------------- | ------------------------ |
| 小型标量捕获（数字、布尔值） | `move`（零拷贝开销）            |
| 大型结构体捕获        | kernel多次读取用 `move`；很少访问用 HMM |
| 原型设计           | 两者皆可；`move` 更可移植         |
| 主机和设备间的共享可变状态  | 引用（HMM）—— 但需注意同步         |


> **提示**
> 
> 不确定时，使用 `move` 闭包。它们更容易推理，到处都能工作，且避免了共享主机/设备内存的同步隐患。

## kernel内闭包

完全在设备代码中定义和调用的闭包以正常 Rust 语义工作 —— 不涉及主机/设备 ABI，因为所有内容已经在 GPU 上：

```rust
#[kernel]
pub fn apply_transform(input: &[f32], mut out: DisjointSlice<f32>) {
    let idx = thread::index_1d();

    let transform = |x: f32| -> f32 {
        let clamped = if x < 0.0 { 0.0 } else if x > 1.0 { 1.0 } else { x };
        clamped * clamped
    };

    if let Some(out_elem) = out.get_mut(idx) {
        *out_elem = transform(input[idx.get()]);
    }
}
```

kernel内闭包由编译器内联，零开销。它们用于在kernel内部提取逻辑，而无需引入单独的设备函数。

## 跨 crate kernel

kernel可以在库 crate 中定义，并从二进制 crate 中启动：

```rust
// 在库 crate `my_kernels` 中：
use cuda_device::{cuda_module, kernel, thread, DisjointSlice};

#[cuda_module]
pub mod kernels {
    use super::*;

    #[kernel]
    pub fn vecadd(a: &[f32], b: &[f32], mut c: DisjointSlice<f32>) {
        let idx = thread::index_1d();
        let i = idx.get();
        if let Some(c_elem) = c.get_mut(idx) {
            *c_elem = a[i] + b[i];
        }
    }
}
```

```rust
// 在二进制 crate 中：
use my_kernels::kernels;

let module = kernels::load(&ctx)?;
module
    .vecadd(&stream, config, &a, &b, &mut c)
    .expect("启动失败");
```

编译器通过 `#[kernel]` 生成的标记 trait 处理跨 crate kernel发现。类型化模块在编译时解析 PTX 名称，并缓存加载的函数句柄。

> **提示**
> 
> 对于泛型跨 crate kernel，单态化发生在**调用** crate 中（具体类型已知的地方），因此 PTX 作为二进制文件编译的一部分生成。
