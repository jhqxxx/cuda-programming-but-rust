# 安全模型 — cuda-oxide

GPU kernel运行数千个线程，它们同时看到同一块内存。在 CPU 上，Rust 通过所有权和借用防止数据竞争 —— 一个可变引用，无别名，在编译时强制执行。在 GPU 上，每个 SM 有 2048 个线程，都从同一个函数启动，都指向同一个输出缓冲区。借用检查器不是为这种场景设计的。

cuda-oxide 分层解决这个问题。常见情况 —— 一个线程写一个元素 —— 按构造就是安全的，不需要 `unsafe`。不常见的情况 —— 共享内存、线程束洗牌、硬件内联函数 —— 需要带文档化合约的 `unsafe`。前沿情况 —— TMA、张量核心、集群级通信 —— 完全手动控制，与它们控制的硬件复杂度相匹配。

本章解释该模型，逐层讲解，并准确告诉你何时需要 `unsafe` 以及为什么。

---

## 三层

cuda-oxide 根据编译器能验证的程度，将kernel安全组织为三层：

| 层级 | 描述 | 是否需要 unsafe？ |
|:---|:---|:---|
| Tier 1 | 安全由构造保证——类型系统阻止误用 | 否 |
| Tier 2 | 显式 unsafe，附带清晰的安全契约 | 是，局部使用 |
| Tier 3 | 原始硬件指令——完全由用户负责 | 是，广泛使用 |

大多数应用kernel完全生活在 Tier 1 或跨越 Tier 1 和 2。Tier 3 面向构建 CUTLASS 或 Triton IR 级别的性能工程师。如果你正在编写向量加法、GEMM 或归约，你很少会离开 Tier 2。

---

## Tier 1：默认安全

### 核心思想：`DisjointSlice<T>` + `ThreadIndex`

主要的安全抽象是一对类型，它们共同保证无竞争的并行写入，在调用点无需 `unsafe`：

- **`ThreadIndex<'kernel, IndexSpace>`** —— 围绕 `usize` 的透明见证 (witness)，没有公开构造函数。获取它的唯一方式是通过受信任的函数（`index_1d`、`index_2d::<S>`），这些函数从**硬件内置变量**（`threadIdx`、`blockIdx`、`blockDim`）派生值 —— 运行时启动时填充的只读特殊寄存器。`ThreadIndex` 是 `!Send + !Sync + !Copy + !Clone`，且 `'kernel` 生命周期绑定到kernel体内部的栈局部作用域，因此它不能通过共享内存走私到别的线程，也不能比kernel活得更久。

- **`DisjointSlice<T, IndexSpace>`** —— 一个切片类型，其 `get_mut()` 方法只接受 `IndexSpace` 与其自身匹配的 `ThreadIndex`。返回 `Option<&mut T>` —— 越界索引返回 `None`。

将它们组合起来，你就得到了一个零 `unsafe` 的kernel：

```rust
use cuda_device::{kernel, DisjointSlice};

#[kernel]
pub fn vecadd(a: &[f32], b: &[f32], mut c: DisjointSlice<f32>) {
    if let Some((c_elem, idx)) = c.get_mut_indexed() {
        let i = idx.get();
        *c_elem = a[i] + b[i];
    }
}
```

`get_mut_indexed` 是一步到位的调用形式：它铸造每个线程的见证，并一次性解析为 `&mut T`。当你需要对多个切片进行并行算术运算时，显式的两步形式也可用：

```rust
#[kernel]
pub fn vecadd(a: &[f32], b: &[f32], mut c: DisjointSlice<f32>) {
    let idx = thread::index_1d();
    if let Some(c_elem) = c.get_mut(idx) {
        let i = idx.get();
        *c_elem = a[i] + b[i];
    }
}
```

安全来自四个事实：

1. `index_1d()` 为每个线程产生唯一值（硬件保证：`threadIdx.x < blockDim.x`，因此线性索引 `blockIdx.x * blockDim.x + threadIdx.x` 在整个网格中唯一）。
2. `get_mut()` 有边界检查 —— 越界线程得到 `None`。
3. `IndexSpace` 参数将每个见证绑定到其布局：`DisjointSlice<T, Index2D<128>>` 不会接受 `ThreadIndex<'_, Index2D<256>>` —— 混合步幅是编译错误。
4. 见证是 `!Send + !Sync + !Copy + !Clone` 且 `'kernel` 作用域的，因此线程不能通过共享内存互相传递索引。

借用检查器看到每个线程有一个 `&mut T`。硬件保证索引是不相交的。类型系统将两者绑定在一起。

### 受信任的索引函数

`ThreadIndex` 的可信度取决于创建它的函数。以下是 cuda-oxide 提供的构造函数：

| 函数 | 公式 | 返回类型 | 说明 |
|:---|:---|:---|:---|
| `index_1d()` | `blockIdx.x * blockDim.x + threadIdx.x` | `ThreadIndex<'kernel, Index1D>` | 每个线程无条件唯一 |
| `index_2d::<S>()` | `row * S + col` | `Option<<ThreadIndex<'kernel, Index2D<S>>>` | 常量步长；混用不同步长会导致编译错误 |
| `unsafe index_2d_runtime(s)` | `row * s + col` | `Option<<ThreadIndex<'kernel, Runtime2DIndex>>` | 调用者需断言所有线程使用了相同的 `s` |
| `index_2d_row()` | `blockIdx.y * blockDim.y + threadIdx.y` | `usize` | 分量访问器，非见证构造器 |
| `index_2d_col()` | `blockIdx.x * blockDim.x + threadIdx.x` | `usize` | 分量访问器，非见证构造器 |

`index_2d_row()` 和 `index_2d_col()` 返回纯 `usize` —— 它们给你用于算术的分量，但不能用于索引 `DisjointSlice`。只有线性化结果才能获得 `ThreadIndex`。

### `index_2d` 如何做到类型安全

`index_2d::<S>()` 在行步幅上是常量泛型的。见证以 `ThreadIndex<'kernel, Index2D<S>>` 返回，而 `DisjointSlice<T, Index2D<S>>` 只接受完全相同的 `S`。两个线程不能用不同步幅铸造见证并传入同一个切片 —— 类型系统拒绝它：

```rust
#[kernel]
pub fn ok(mut out: DisjointSlice<u32, Index2D<128>>) {
    if let Some(idx) = thread::index_2d::<128>() {  // 匹配
        if let Some(slot) = out.get_mut(idx) { *slot = 1; }
    }
}

#[kernel]
pub fn rejected(mut out: DisjointSlice<u32, Index2D<128>>) {
    if let Some(idx) = thread::index_2d::<256>() { // ⛔ Index2D<256> != Index2D<128>
        if let Some(slot) = out.get_mut(idx) { *slot = 1; }
    }
}
```

见证也是 `!Send + !Sync + !Copy + !Clone`，其 `'kernel` 生命周期从宏注入的栈局部作用域借用 —— 因此线程不能将其 `ThreadIndex` 存放在共享内存中让邻居稍后拾取，且见证不能比kernel体活得更久。

#### 真正的运行时步幅：`unsafe index_2d_runtime`

某些kernel确实在启动时接收步幅（例如仅在主机上知道的矩阵维度）。对于这些情况，有对应的见证 `Runtime2DIndex` 和一个 `unsafe` 构造函数：

```rust
let idx = unsafe { thread::index_2d_runtime(n)? };
```

这里的 `unsafe` 是合约：kernel中每个将 `Runtime2DIndex` 传入同一个 `DisjointSlice<T, Runtime2DIndex>` 的线程必须使用相同的 `n`。类型系统*无法*证明这一点 —— 在不同运行时步幅下产生的两个 `ThreadIndex<'_, Runtime2DIndex>` 值具有相同类型。如果你能在编译时固定步幅，优先使用 `index_2d::<S>()`。如果不能，`index_2d_runtime` 上的 `unsafe` 关键字就是安全义务归你的标记。

### GEMM 模式

对于常量步幅的二维kernel（常见情况 —— 分块 GEMM、模板kernel、固定通道数的图像kernel），常量泛型形式是安全的默认选择：

```rust
const STRIDE: usize = 1024;     // C 是 M x STRIDE

#[kernel]
pub fn gemm(a: &[f32], b: &[f32], mut c: DisjointSlice<f32, Index2D<STRIDE>>, m: u32) {
    let row = thread::index_2d_row();
    if let Some((c_elem, _)) = c.get_mut_indexed() {
        // col < STRIDE 由 `Some` 保证 —— 无需手动检查
        if row < m as usize {
            // ... 将点积计算到局部累加器 `sum` ...
            *c_elem = alpha * sum + beta * (*c_elem);
        }
    }
}
```

对于运行时步幅，相同的结构也适用，但线性化步骤是显式且 `unsafe` 的：

```rust
#[kernel]
pub fn gemm_runtime(a: &[f32], b: &[f32], mut c: DisjointSlice<f32, Runtime2DIndex>, m: u32, n: u32) {
    let n = n as usize;             // 一个绑定，一个步幅值
    let row = thread::index_2d_row();

    // SAFETY: kernel中每个线程看到相同的 `n`（kernel参数）。
    if let Some(c_idx) = unsafe { thread::index_2d_runtime(n) } {
        if row < m as usize {
            // ... 计算点积 ...
            if let Some(c_elem) = c.get_mut(c_idx) {
                *c_elem = alpha * sum + beta * (*c_elem);
            }
        }
    }
}
```

`index_2d_runtime` 的 `if let Some` 替代了你在 CUDA C++ 中写的 `col < n` 手动守卫。`row < m` 检查保留，因为它防止从输入矩阵读取垃圾数据。

### 什么使kernel成为 Tier 1

当满足以下条件时，kernel是完全安全的 —— Tier 1：

1. 所有可变输出访问通过 `DisjointSlice::get_mut(ThreadIndex)`
2. 所有输入是共享不可变引用（`&[T]`）
3. 无共享内存、无原始指针、无线程索引之外的内联函数

此层级中的示例包括 `vecadd`、`helper_fn`、`generic`、`host_closure`，以及 `gemm` 和 `async_mlp` 示例中的朴素 GEMM kernel。

## Tier 2：作用域 `unsafe`

并非每个kernel都适合"一个线程，一个输出元素"的模式。当线程需要协作时 —— 通过快速片上内存共享数据、在线程束的通道间通信，或执行原子更新 —— 你需要 `unsafe`。Tier 2 的关键属性是 `unsafe` 是*作用域的*和*可审计的*：每个块都有文档化的安全合约，kernel的其余部分保持安全。

### 共享内存

共享内存快速、片上，且对块内每个线程可见。最后一个属性正是它需要 `unsafe` 的原因 —— 借用检查器无法推理 256 个线程写入同一个 `static mut` 数组：

```rust
static mut TILE: SharedArray<f32, 256> = SharedArray::UNINIT;

unsafe { TILE[ty * TILE_SIZE + tx] = value; }

thread::sync_threads();

let neighbor = unsafe { TILE[other_idx] };
```

合约：确保没有同步就没有并发线程的冲突写入。`sync_threads()` 屏障是使其工作的工具 —— 它保证所有线程完成写入后，任何线程才开始读取。

| API | 安全义务 |
|:---|:---|
| `SharedArray<T, N>` | 通过 `static mut` 访问。未同步时不得出现冲突写入。 |
| `DynamicSharedArray<T>` | 规则同上，但大小在启动时通过 `LaunchConfig::shared_mem_bytes` 设置。 |

### 线程束内联函数

线程束级原语让线程束内的线程在不接触内存的情况下交换数据 —— 寄存器到寄存器传输，由硬件协调。它们是 `unsafe` 的，因为硬件不检查线程收敛：如果你传递的掩码包含已分歧的线程，你会得到未定义行为（通常是静默挂起，这比崩溃更糟）。

| API | 安全义务 |
|:---|:---|
| `shfl_sync`, `shfl_up_sync`, `shfl_down_sync`, `shfl_xor_sync` | 源线程必须处于活跃状态；掩码必须包含调用线程 |
| `ballot_sync`, `any_sync`, `all_sync` | 掩码中的所有线程必须已收敛 |
| `activemask` | 结果仅在调用点有意义 |

### 屏障和生命周期

`ManagedBarrier` 类型状态 API 将屏障生命周期（`Uninit` -> `Ready` -> `Invalidated`）编码在类型系统中，因此你不能等待从未初始化的屏障，或使用已失效的屏障。`init()` 和 `inval()` 转换仍然需要 `unsafe`，因为它们与硬件交互，但类型状态在编译时防止最常见的错误。

| API | 安全义务 |
|:---|:---|
| `mbarrier_init` | 必须由恰好一个线程调用；屏障必须位于共享内存中 |
| `mbarrier_arrive` / `mbarrier_wait` | 屏障必须已完成初始化；令牌必须匹配 |
| `ManagedBarrier`（类型状态） | `init()` 和 `inval()` 需要 `unsafe`；状态机在编译期强制执行 |

### 原子操作

一旦你拥有有效的原子引用，原子操作调用是安全的。`unsafe` 表面在构造时 —— 从原始指针创建 `DeviceAtomicU32` 需要调用者保证指针有效且正确对齐：

```rust
let atom = unsafe { DeviceAtomicU32::new(ptr) };
atom.fetch_add(1, Ordering::Relaxed);  // 安全调用
```

### 无检查切片访问

当"一个线程，一个元素"模型不适用时 —— 例如在线程束级归约中只有 lane 0 写入结果 —— `DisjointSlice::get_unchecked_mut(usize)` 提供逃生舱：

```rust
if warp::lane_id() == 0 {
    let warp_idx = gid.get() / 32;
    // SAFETY: 每个线程束只有 lane 0 写入；线程束索引唯一
    unsafe { *out.get_unchecked_mut(warp_idx) = sum; }
}
```

安全义务与 `ThreadIndex` 系统自动强制执行的相同：索引在边界内，没有两个线程共享同一索引。区别在于你自己证明它，而不是让类型系统为你做。

## Tier 3：原始硬件

在栈底是原始硬件内联函数 —— 直接与特定 GPU 架构上特定功能单元对话的 API。每次调用都是 `unsafe`，安全合约复杂且架构相关，文档更多存在于 PTX ISA 手册中而非 Rust 文档注释里。

| 特性 | 关键 API | 架构 |
|:---|:---|:---|
| TMA（张量内存加速器） | `tma_load_2d`, `tma_store_2d`, `TmaDescriptor` | sm_90+ (Hopper) |
| tcgen05（第五代张量核心） | `tcgen05_mma`, `tcgen05_commit`, `TensorMemoryHandle` | sm_120 (Blackwell) |
| WGMMA（线程组 MMA） | `wgmma_mma_async`, `wgmma_commit_group`, `wgmma_wait_group` | sm_90+ (Hopper) |
| Cluster（集群） | `cluster_rank`, `map_shared_rank`, `cluster_barrier_arrive` | sm_90+ (Hopper) |
| CLC（集群启动控制） | `clc_prefetch`, `clc_query_channel` | sm_120 (Blackwell) |
| TMEM（张量内存） | `TmemGuard`（类型状态）, `tmem_alloc`, `tmem_dealloc` | sm_120 (Blackwell) |

如果你正在编写应用级kernel，你不需要 Tier 3 API。它们面向构建下一个 CUTLASS 的人 —— 对于这些人，cuda-oxide 提供与 CUDA C++ 内联 PTX 相同的硬件访问，Rust 的类型系统可用（但不强制执行）作为护栏。

## 借用检查器给了你什么

cuda-oxide 不是 DSL 或宏系统 —— 它在你的kernel代码上运行真正的 `rustc` 前端。这意味着 Rust 在 CPU 上提供的每个安全保证也在 GPU 上强制执行：

| 保证 | 工作原理 |
|:---|:---|
| 所有权与借用 | 生命周期错误、释放后使用、别名违规在编译期捕获 |
| 安全的并行写入 | `DisjointSlice<T>` + `ThreadIndex` —— 在类型层面证明写入不会竞争 |
| 显式 unsafe 作用域 | 原始指针访问需要 `unsafe`，使安全义务可见且可审计 |
| convergent 属性强制 | 同步原语（屏障、栅栏、洗牌操作）在 IR 中标记为 convergent，防止优化器在控制流间移动或复制它们 |

前三个是标准 Rust。第四个是 GPU 特有的：CUDA 的 `bar.sync`、fence 和线程束洗牌指令不能被编译器复制或重排序。cuda-oxide 在 IR 中将它们标记为 `convergent`，以便 LLVM 的优化遍不触碰它们。

## 难题

Rust 的借用检查器是为单线程所有权设计的，用 `Send`/`Sync` 处理 CPU 并发。SIMT 执行引入了借用检查器从未学过推理的模式。以下是 cuda-oxide 今天*不*强制执行的诚实清单 —— 以及为什么这些问题是可解决的。

### 线程分歧控制流

Rustc 的 JumpThreading MIR 优化将函数调用复制到 if 语句的两个分支中 —— 在 CPU 上是完全合理的优化，但它破坏了 GPU 屏障语义，其中块中所有线程必须在同一个 `bar.sync` 指令处汇合。cuda-oxide 目前为设备代码禁用 JumpThreading（`-Z mir-enable-passes=-JumpThreading`）。正确的解决方案是教会编译器收敛要求，使其能够围绕它们优化，而不是完全禁用该遍。

### 共享内存访问模式

借用检查器无法推理线程 0 写入 `smem[0]` 和线程 1 写入 `smem[1]` 是否安全 —— 它看到 `&mut smem` 并拒绝。`DisjointSlice` 解决了唯一索引写入模式，但无法解决协作模式如归约、扫描或生产者/消费者流水线，其中多个线程在阶段之间有同步的情况下有意访问重叠区域。

### 线程束级收敛

像 `shfl_sync` 和 `ballot_sync` 这样的操作要求参与掩码中命名的所有线程在调用点实际收敛。类型系统今天无法强制执行这一点。如果线程已分歧而你传递完整掩码，你会得到静默挂起 —— 最糟糕的 bug 类型，因为没有崩溃、没有错误消息，只有永不完成的kernel。

### 内存空间感知

GPU 内存有不同的地址空间 —— 全局、共享、局部、TMEM。指向共享内存的 `&mut` 对块内每个线程可见；指向局部内存的 `&mut` 对一个线程私有。借用检查器对它们一视同仁。这是保守的（它拒绝一些安全程序）但从不错误（它不会接受不安全的程序）。尽管如此，内存空间感知的借用检查器可以在不需要 `unsafe` 的情况下接受更多程序。

### 为什么这些是可解决的

构建块已经存在于 Rust 的类型系统中。它们需要扩展，而非重新发明：

| 想法 | 解决的问题 |
|:---|:---|
| 执行资源感知类型 | 函数标注其执行层级（grid / block / warp / thread）。在发散分支内调用屏障将成为编译期错误。 |
| 内存视图 | 泛化的并行访问模式——类似于 `DisjointSlice`，但覆盖分块、条带化、转置及组合布局。在规模上实现类型检查的竞态自由。 |
| 同步的扩展借用检查 | 静态强制屏障不能被遗忘、放置在发散控制流中，或被优化器复制。在类型系统中实现收敛性。 |

所有这些都是编译时分析。生成的 PTX 与你手写的一致 —— 安全网在代码生成时消失。零运行时成本。

cuda-oxide 有能力逐步交付这些。真正的 `rustc` 借用检查器已经在设备代码上运行。IR 基础设施（pliron dialects）支持 GPU 感知分析遍。从 MIR 到 PTX 的完整编译流水线在我们的控制之下。每个新的安全检查都是增量式的 —— 现有kernel继续编译，新kernel获得更强的保证。

## 编写安全kernel：速查表

### 默认路径

对于大多数kernel，从这里开始：

```rust
#[kernel]
pub fn my_kernel(input: &[f32], mut output: DisjointSlice<f32>) {
    if let Some((out, idx)) = output.get_mut_indexed() {
        *out = transform(input[idx.get()]);
    }
}
```

对于常量步幅二维，参数化切片并请求常量索引：

```rust
#[kernel]
pub fn tile_kernel(mut output: DisjointSlice<f32, Index2D<1024>>) {
    if let Some((out, _idx)) = output.get_mut_indexed() {
        *out = ...;
    }
}
```

规则：

- 所有可变输出使用 `DisjointSlice`。
- 所有只读输入使用 `&[T]`。
- 对于一维网格，默认使用 `get_mut_indexed()`。如果你需要对多个切片进行算术运算，回退到显式对：`let idx = thread::index_1d(); slice.get_mut(idx)`。
- 对于常量步幅二维网格，将切片参数化为 `DisjointSlice<T, Index2D<S>>`，使用 `get_mut_indexed()` 或 `thread::index_2d::<S>()`。不匹配的步幅是编译错误。
- 对于运行时步幅，使用 `unsafe { thread::index_2d_runtime(n) }` 配合 `Runtime2DIndex` 标记的切片。`unsafe` 是合约，即每个线程使用相同的 `n`。
- 始终通过 `get_mut()` / `get_mut_indexed()` 进行边界检查（两者都返回 `Option`）。

如果你的kernel编译时没有 `unsafe`，它按构造就是无竞争的。

### 何时需要 `unsafe`

| 模式 | 原因 | 缓解措施 |
|:---|:---|:---|
| 共享内存 | 多个线程访问同一个 `static mut` | 跨线程读取前使用 `sync_threads()` 同步 |
| Warp 洗牌 | 线程收敛性未经编译器检查 | 全 Warp 操作使用 `FULL_MASK`；部分掩码需文档说明 |
| 原子操作 | 由原始指针构造 | 封装到辅助函数中；原子操作本身是安全的 |
| 非均匀写入 | 并非每个线程都写入自己的索引 | 使用 `get_unchecked_mut` 并附上唯一性参数的文档说明 |
| 硬件指令 | 复杂、架构相关的契约 | 遵循 PTX ISA 文档；在目标硬件上测试 |

## SAFETY 注释

对于每个 `unsafe` 块，都要记录**为什么**不变量成立。不是代码做了什么——代码本身已经说明了这一点——而是**为什么**这种特定用法是安全的：

```rust
// SAFETY: 只有每个 warp 的 lane 0 执行这个分支。
// Warp 索引 (gid / 32) 在不同 warp 之间是唯一的，
// 因此不会有两个线程写入同一个输出元素。
if warp::lane_id() == 0 {
    let warp_idx = gid.get() / 32;
    unsafe { *partial_sums.get_unchecked_mut(warp_idx) = warp_sum; }
}
```

这不是仪式。当凌晨 2 点kernel发生数据竞争，你盯着 `compute-sanitizer` 的日志时，过去的你留下的安全注释是定位 bug 的最快途径。

> **提示**
> 
> 如果你无法为一个 `unsafe` 块写出令人信服的 SAFETY 注释，这就是一个信号：该不变量实际上并未被维护。重构代码直到论证显而易见，或者改用安全的 API。



## 总结

| 属性 | 状态 |
|:---|:---|
| 设备代码上的借用检查器 | 已强制执行（真实的 rustc 前端） |
| 安全的 1D 并行写入（`DisjointSlice` + `index_1d`） | 已强制执行 |
| 安全的 2D 并行写入——常量步长 | 已强制执行（`Index2D<S>` 不匹配会导致编译错误） |
| 安全的 2D 并行写入——运行时步长 | 通过 `unsafe index_2d_runtime` 由调用者断言 |
| `ThreadIndex` 不可跨线程传递（共享内存清洗） | 已强制执行（`!Send + !Sync + !Copy + !Clone + 'kernel`） |
| `&mut [T]` kernel参数 | **未**强制执行——将任何 `&mut` 参数视为 `unsafe` |
| 共享内存、指令的显式 `unsafe` | 已强制执行（Rust 语言规则） |
| 同步原语上的 `convergent` 属性 | 已强制执行（IR 级 convergent 标记） |
| Warp 操作的线程收敛性 | **未**强制执行（运行时义务） |
| 内存空间感知（共享内存 vs 全局内存） | **未**强制执行（未来工作） |

`&mut [T]` 作为kernel参数是下一个突出的缺口：宏今天接受该类型，但运行时布局（每个线程看到相同的支持指针）意味着两个不同线程通过 `&mut data[i]` 写入，正是 `DisjointSlice` 存在以防止的那种别名。在宏直接拒绝 `&mut [T]`（或将其重写为 `DisjointSlice`）之前，将任何接受它的kernel视为每一行都是 `unsafe`。

安全模型旨在使常见情况默认安全，同时为其他一切提供显式逃生舱。编写你的kernel，让类型系统捕获竞争，将 `unsafe` 留给那些你确实知道编译器不知道的事情。
