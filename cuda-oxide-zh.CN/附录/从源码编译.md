# 从源码构建

本附录将指导您从一个全新的代码仓库克隆开始，设置 cuda-oxide 的开发环境。如果您只是想运行一个示例，[编写您的第一个kernel](../getting-started/hello-gpu.md) 一章会更快。

---

## 环境要求

| 依赖项   | 版本      | 用途       |
|:------|:---------|:-------------|
| **Rust nightly** | `nightly-2026-04-03` (固定版本) | 带有 `rustc-dev` 的编译器工具链，用于代码生成后端          |
| **CUDA Toolkit** | 12.x+   | 驱动 API、`nvcc`、PTX 汇编器     |
| **LLVM**    | 21+ 且包含 NVPTX 后端  | `llc-21`/`llc-22`，用于 LLVM IR → PTX 降级    |
| **Clang**  | 21+ (`clang-21` 包)   | 主机端 `cuda-bindings` 中的 `bindgen` 需要 clang 的头文件 |
| **Linux**  | 已在 Ubuntu 24.04 上测试   | 不支持 Windows 和 macOS     |
| **GPU**  | sm_80, sm_90, sm_100a    | 硬件目标     |

## 克隆仓库

```bash
git clone https://github.com/NVlabs/cuda-oxide.git
cd cuda-oxide
```

## 安装 Rust 工具链

仓库中附带了一个 `rust-toolchain.toml` 文件，固定了确切的 nightly 版本和组件。Rustup 会自动识别它：

```toml
# rust-toolchain.toml (已在仓库根目录)
[toolchain]
channel = "nightly-2026-04-03"
components = ["rust-src", "rustc-dev", "rust-analyzer"]
```

如果需要手动安装：

```bash
rustup toolchain install nightly-2026-04-03
rustup component add rust-src rustc-dev --toolchain nightly-2026-04-03
```

`rust-src` 提供交叉编译所需的标准库源代码，`rustc-dev` 暴露代码生成后端所依赖的编译器内部接口。

## 安装 CUDA

确保 CUDA 工具链在您的 `PATH` 中：

```bash
export PATH="/usr/local/cuda/bin:$PATH"
nvcc --version   # 应打印 12.x 或更高版本
```

如果您在没有 GPU 的系统上构建（例如 CI），仍然需要工具链中的 `ptxas` 和头文件，但无法运行kernel。

## 安装 LLVM

代码生成流水线会发出 LLVM IR 并调用 `llc` 来生成 PTX。需要 LLVM 21 或更高版本 —— 较早的 `llc` 版本会拒绝 cuda-oxide 发出的 TMA / tcgen05 / WGMMA 内联函数签名。必须启用 NVPTX 后端：

```bash
# Ubuntu / Debian
sudo apt install llvm-21
```

如果您的发行版软件包未提供 `llvm-21`，请使用 LLVM 的 apt 辅助脚本：

```bash
sudo apt-get install -y lsb-release wget software-properties-common gnupg
wget https://apt.llvm.org/llvm.sh && chmod +x llvm.sh
sudo ./llvm.sh 21
```

```bash
# 验证 NVPTX 支持
llc-21 --version | grep nvptx
```

您应该在目标列表中看到 `nvptx64 - NVIDIA PTX 64-bit`。流水线会自动发现 `PATH` 中的 `llc-22` 和 `llc-21`（按此顺序）；要固定特定二进制文件，请设置 `CUDA_OXIDE_LLC=/usr/bin/llc-21`。

> **注意：**
> 较旧的 `llc` 二进制文件（LLVM 20 及更早）在通过 `CUDA_OXIDE_LLC=/path/to/llc-20` 指定时，可以编译较简单的kernel，但任何使用现代 TMA / tcgen05 / WGMMA 内联函数（`tma_copy`、`gemm_sol`、`tcgen05_matmul`、`wgmma`，……）的示例都会失败，并报错 `Intrinsic has incorrect argument type!`，直到您升级到 LLVM 21+。

## 安装 Clang（用于主机端 `cuda-bindings`）

主机端 `cuda-bindings` crate 会运行 `bindgen`，它会加载 libclang 并需要 clang 自己的资源目录中的 `stddef.h` —— 仅安装 `libclang1-*` 运行时是不够的。

```bash
sudo apt install clang-21   # 或 libclang-common-21-dev
```

`cargo oxide doctor` 会预先验证这一点。

## 构建工作区

主工作区包含面向用户的 crate（`cuda-device`、`cuda-core`、`cuda-async` 等）和构建工具（`cargo-oxide`）：

```bash
cargo build
```

> **注意：**
> 代码生成后端（`crates/rustc-codegen-cuda/`）**故意不**作为工作区成员，因为它需要特殊的 nightly 特性和不同的构建过程。`cargo-oxide` 会透明地处理其构建。

## 安装 cargo-oxide

`cargo-oxide` 是驱动完整编译流水线的 cargo 子命令。在仓库内部，它通过工作区别名工作。对于独立使用：

```bash
cargo install --git https://github.com/NVlabs/cuda-oxide.git cargo-oxide
```

首次运行时，`cargo-oxide` 会自动获取并构建代码生成后端动态库。

## 验证安装

```bash
# 检查所有前置条件
cargo oxide doctor

# 编译并运行第一个标准示例
cargo oxide run vecadd
```

`cargo oxide doctor` 会验证您的 Rust 工具链、CUDA 工具包（包括用于使用数学内联函数的kernel的 libNVVM / nvJitLink / libdevice）、LLVM 安装和代码生成后端。如果所有配置都正确，`cargo oxide run vecadd` 会将一个 Rust kernel编译为 PTX，在 GPU 上启动它，并打印成功消息。

## 常用命令

```bash
# 构建并运行一个示例
cargo oxide run <example>

# 显示完整编译流水线（MIR → LLVM IR → PTX）
cargo oxide pipeline <example>

# 使用 cuda-gdb 调试
cargo oxide debug <example> --tui

# 构建用于 libNVVM/nvJitLink 互操作的 NVVM IR
cargo oxide build <example> --emit-nvvm-ir --arch sm_120
```

## 构建本书

文档位于 `cuda-oxide-book/` 中，使用 Sphinx 和 MyST Markdown。要在本地构建并提供服务：

```bash
cd cuda-oxide-book
make setup      # 创建虚拟环境，安装依赖
source .venv/bin/activate
make livehtml   # 在 http://localhost:8000 启动开发服务器
```

## 生成 API 文档

标准 `cargo doc` 可用于工作区中的 crate：

```bash
cargo doc --no-deps --open
```

这将为 `cuda-device`、`cuda-core`、`cuda-async` 以及所有其他工作区成员生成 rustdoc。代码生成后端由于不是工作区成员而被排除在外。

## 工作区结构

```text
cuda-oxide/
├── Cargo.toml              # 工作区根目录
├── rust-toolchain.toml     # 固定的 nightly 版本和组件
├── crates/
│   ├── cuda-device/          # 设备内联函数 (#![no_std])
│   ├── cuda-host/            # 主机启动 API
│   ├── cuda-macros/          # 过程宏 (#[kernel], #[device], gpu_printf!)
│   ├── cuda-bindings/        # 到 cuda.h 的原始 bindgen FFI
│   ├── cuda-core/            # 安全的 RAII 包装器 (CudaContext, DeviceBuffer)
│   ├── cuda-async/           # 异步执行 (DeviceOperation, DeviceFuture)
│   ├── cargo-oxide/          # Cargo 子命令
│   ├── rustc-codegen-cuda/   # 代码生成后端（不是工作区成员）
│   ├── mir-importer/         # MIR → Pliron IR 翻译
│   ├── mir-lower/            # `dialect-mir` → `dialect-llvm` 降级
│   ├── dialect-mir/          # 建模 Rust MIR 的 pliron dialect
│   ├── dialect-llvm/         # 建模 LLVM IR 的 pliron dialect（+ 导出）
│   ├── dialect-nvvm/         # NVVM 内联函数dialect
│   ├── libnvvm-sys/          # libNVVM 的 dlopen 绑定
│   ├── nvjitlink-sys/        # nvJitLink 的 dlopen 绑定
│   ├── reserved-oxide-symbols/ # 共享命名约定
│   └── fuzzer/               # 差异测试支持
└── cuda-oxide-book/        # 本书（Sphinx + MyST）
```

## 故障排除

`llc` 未找到或缺少 NVPTX
: 确保安装了 LLVM 21+，并且 `llc-21`（或 `llc-22`）在您的 `PATH` 中。流水线按顺序探测 `llc-22` → `llc-21`；或者设置 `CUDA_OXIDE_LLC=/usr/bin/llc-21` 以指向特定二进制文件。

`Intrinsic has incorrect argument type!`（来自 `llc`）
: 您的 `llc` 版本低于 LLVM 21，无法降级现代 TMA / tcgen05 / WGMMA 内联函数签名。请安装 `llvm-21` 并重新运行。

`error[E0463]: can't find crate for rustc_middle`
: 您缺少 `rustc-dev` 组件。请运行：
  `rustup component add rustc-dev --toolchain nightly-2026-04-03`。

CUDA 驱动程序版本不匹配
: 工具包版本（编译时）和驱动程序版本（运行时）必须兼容。运行 `nvidia-smi` 检查驱动程序版本，运行 `nvcc --version` 检查工具包版本。

`cargo oxide doctor` 在代码生成后端失败
: 后端在首次使用时构建。如果构建失败，请检查是否安装了 `rust-src`，以及 nightly 版本是否与 `rust-toolchain.toml` 匹配。
