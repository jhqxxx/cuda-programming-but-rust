# 支持特性

本附录展示了 cuda-oxide 的特性矩阵：每个编译器能力、运行时 API 和硬件特性及其当前支持状态。数据来自编译器/运行时源码和测试套件。

**图例：** **完全** = 已测试且正常工作，**部分** = 已实现且可用但存在已知缺口（在行描述中说明），**计划中** = 在路线图上，**不适用** = 不适用或未识别出需求。

---

## 编译器：内存模型

|特性|状态|描述|
|:---|:---|:---|
|HMM / 统一内存管理|**完全**|GPU 直接读写主机内存，无需 `cudaMemcpy`。闭包中的引用捕获利用 HMM 访问主机指针。需要 Turing+ GPU、Linux 6.1.24+、CUDA 12.2+。|
|统一结构体 ABI（无需 `#[repr(C)]`）|**完全**|设备端结构体布局与主机完全匹配。编译器查询 rustc 的实际布局，并在 LLVM IR 中使用显式填充重现。适用于默认的 `#[repr(Rust)]`。|
|动态布局匹配|**完全**|编译器查询 rustc 的 `fields_by_offset_order()` 和字节偏移量，构建具有正确字段顺序和显式填充字节的 LLVM 结构体。独立于 LLVM 的 datalayout。|

## 编译器：类型系统

|特性|状态|描述|
|:---|:---|:---|
|泛型与单态化|**完全**|带有 trait 边界的泛型kernel和设备函数。从 rustc MIR 收集单态化实例。支持常量泛型。|
|枚举（`Option<T>`、`Result<T,E>`、自定义）|**完全**|完整枚举支持，包括判别式提取和有效负载访问。支持枚举上的模式匹配。|
|结构体构造与字段访问|**完全**|结构体字面量、字段访问、按值传递和返回值。支持用户定义的结构体，无需注解。|
|数组类型（`[T; N]`）|**完全**|静态数组构造、常量索引和运行时索引访问。可变数组自动提升为内存支持。|
|`CuSimd<T, N>` SIMD 类型|**完全**|泛型 SIMD 寄存器类型，带有命名访问器（`x`/`y`/`z`/`w`）、运行时和编译时索引、`to_array` 转换。|
|ABI 标量化|**完全**|切片在kernel边界处被标量化（`&[T]` -> `(ptr, len)`，在函数内部重建）。结构体和闭包按值作为一个 byval `.param` 传递；在内部设备到设备调用中仍然应用字段展平。|

## 编译器：闭包

|特性|状态|描述|
|:---|:---|:---|
|移动闭包（`FnOnce`）|**完全**|按值捕获的闭包。整个闭包结构体作为一个 byval kernel参数推送。`move \|x\| x * factor` 模式。|
|引用闭包（`Fn`/`FnMut`）|**完全**|通过引用捕获的非移动闭包。闭包结构体（包含主机指针）仍然作为一个 byval 参数传递；GPU 通过 HMM 读取这些指针。|
|主机到设备闭包|**完全**|在主机上定义并传递给泛型kernel的闭包。测试了带捕获系数的多项式求值。|
|设备内部闭包|**完全**|完全在设备上创建和使用的闭包，包括传递给设备函数的闭包。|

## 编译器：控制流

|特性|状态|描述|
|:---|:---|:---|
|匹配表达式（整数分支）|**完全**|整数上的多路匹配。生成条件分支链。|
|枚举上的匹配|**完全**|对 `Option<T>` 和自定义枚举的模式匹配。判别式提取 + 有效负载访问。|
|`for` 循环（区间、迭代器、枚举）|**完全**|完整的迭代器脱糖：基于范围的循环、`slice.iter()`、`enumerate()`、嵌套循环、`break`、`continue`。|
|`while` 循环 / `if-else`|**完全**|基础控制流完全支持。|
|`break` 和 `continue`|**完全**|在 `for`/`while` 循环中的 `break` 和 `continue`，包括提前退出。|

## 编译器：算术与类型转换

|特性|状态|描述|
|:---|:---|:---|
|64 位算术|**完全**|完整的 64 位整数算术，包括移位、位运算和描述符字段打包。|
|类型转换（所有种类）|**完全**|整数到整数、整数到浮点、浮点到整数、浮点到浮点、内存转置（位转换）、指针到指针、指针到整数、整数到指针、指针强制转换。|

## 编译器：互操作

|特性|状态|描述|
|:---|:---|:---|
|双向 LTOIR 支持|**完全**|Rust kernel调用 CUDA C++ 设备函数，且 C++ 调用 Rust 设备函数。通过 NVVM IR → libNVVM → LTOIR → nvJitLink 实现。|
|设备 FFI（`extern "C"`）|**完全**|为外部 LTOIR 函数声明 `#[device] extern "C" { fn ... }`。已演示 CUB/CCCL 集成。|
|MathDx FFI（cuFFTDx / cuBLASDx）|**完全**|通过 LTOIR 实现 cuFFTDx（8/16/32 点线程级 FFT）、cuBLASDx（32x32x32 块级 GEMM）。|
|跨 crate kernel|**完全**|在库 crate 中定义的kernel和设备函数，在二进制 crate 使用点进行单态化。|

## 编译器：函数

|特性|状态|描述|
|:---|:---|:---|
|`#[kernel]` 属性|**完全**|将函数标记为 GPU kernel入口点（`ptx_kernel` 调用约定）。每个文件可以有多个kernel。|
|`#[device]` 辅助函数|**完全**|可从kernel调用的设备端辅助函数。被 `llc` 积极内联。|
|独立的 `#[device]` 函数|**完全**|在没有kernel存在时编译的设备函数。为 C++ 使用提供干净的导出名称。|
|多kernel模块|**完全**|单个源文件中的多个 `#[kernel]` 函数编译为单个 PTX 模块。|

## 编译器：编译流水线

|特性|状态|描述|
|:---|:---|:---|
|统一单源编译|**完全**|主机和设备代码位于同一文件中。自定义 rustc 代码生成后端拦截代码生成。无需 `#[cfg]`。|
|PTX 输出|**完全**|默认输出：Rust MIR → `dialect-mir` → `mem2reg` → `dialect-llvm` → LLVM IR → `llc` → PTX。支持 sm_80 到 sm_100a。|
|NVVM IR 输出|**完全**|用于 libNVVM 消费的备选输出，包含 NVVM 元数据。|
|LTOIR 链接|**完全**|通过 libNVVM 和 nvJitLink 进行设备端 LTO。|
|浮点数学内联函数（libdevice）|**完全**|Rust `f32`/`f64` 数学方法（`sin`、`cos`、`exp`、`pow`、`sqrt`……）降级为 CUDA libdevice（`__nv_*`）。cuda-oxide 自动检测 libdevice 使用并发出 NVVM IR；`cuda_host::load_kernel_module`（同步）和 `cuda_host::load_kernel_module_async`（异步）在运行时通过 libNVVM + nvJitLink 构建 cubin。|
|流水线检查|**完全**|`cargo oxide pipeline <example>` 显示每个编译阶段的 IR。|
|cuda-gdb 调试支持|**完全**|使用调试信息构建并启动 `cuda-gdb`。`breakpoint()` 内联函数用于编程断点。|

## 运行时库：安全性

|特性|状态|描述|
|:---|:---|:---|
|`DisjointSlice<T, IndexSpace>`|**完全**|边界检查的并行写入输出切片。`get_mut` 和 `get_mut_indexed` 返回 `Option<&mut T>`。`IndexSpace` 类型参数在编译时拒绝不匹配的 2D 步长。|
|`ThreadIndex<'kernel, IndexSpace>`|**完全**|不透明见证，只能由可信的索引函数构造。`!Send + !Sync + !Copy + !Clone`，`'kernel` 作用域 —— 不可跨线程传递，不能超过kernel体生命周期。|
|`ManagedBarrier` 类型状态|**完全**|编译时屏障生命周期：`Uninit → Ready → Invalidated`。无效转换是编译错误。|

|块作用域原子操作|**完全**|`BlockAtomic{U32,I32,U64,I64,F32,F64}`，`.cta` 作用域。|

## 尚未实现

|特性|状态|说明|
|:---|:---|:---|
|内联汇编（`asm!` 宏）|**计划中**|替代方案：使用内置内联函数或向 `cuda-device` 添加新的内联函数。|
|FP8 / MX 数据类型|**计划中**|Blackwell 的路线图项。无架构限制。|
|动态分发（`dyn Trait`）|**不适用**|使用带有静态分发的泛型。未发现真正需要此功能。|
|堆分配（`Box`、`Vec`）|**不适用**|CUDA 有一个设备端堆（kernel中的 `malloc`/`free`），编译器允许 `alloc` crate 通过 —— 但目前没有连接设备端 `#[global_allocator]`。即使连接了，设备 `malloc` 也极其缓慢（串行、碎片化、非合并）。请使用切片和 `SharedArray`。|
|`String` / `format_args!`|**不适用**|使用 `gpu_printf!` 进行格式化输出。|
|Panic / 展开|**不适用**|MIR 中存在 panic 路径，但编译器会剥离 `core::panicking::*` 和所有展开边。GPU 硬件*可以*支持展开（绝对分支 + Volta 之后的每线程调用栈跟踪），但 CUDA 工具链（nvcc/ptxas）目前不暴露它 —— 没有着陆垫能存活到 PTX。如果在运行时到达 panic 路径，GPU 会陷入陷阱（等同于 `panic=abort`）。NVIDIA 有一个正在进行的项目，为 CUDA 添加 C++ 异常支持，用于汽车安全；当前 cuda-oxide 的设计与该工作向前兼容。现在请使用 `gpu_assert!()` + `trap()` 进行显式运行时检查。|
|标准库（`std`/`alloc`）|**不适用**|`std` 被禁止。`alloc` 被收集器允许，但没有后备分配器。只有 `core` 完全可用。`Option`、`Result`、迭代器都能工作。|
|纹理内存|**不适用**|考虑到 Hopper+ 上 TMA 的可用性，优先级较低。|
