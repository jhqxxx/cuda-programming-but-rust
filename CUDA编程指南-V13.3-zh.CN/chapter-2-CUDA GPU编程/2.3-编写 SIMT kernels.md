# 2.3. 编写 SIMT 内核

CUDA 内核的编写方式在很大程度上与针对给定问题编写传统 CPU 代码的方式相同。但是，GPU 有一些独特的特性可以用来提高性能。此外，对 GPU 上的线程如何调度、如何访问内存以及如何执行有一定了解，有助于开发人员编写能够最大化利用可用计算资源的内核。

本章将介绍使用 C++ 和 Python 中的 SIMT 编程模型编写内核的更多细节。

## 2.3.1. SIMT 基础

从开发人员的角度来看，CUDA 线程是并行的基本单元。[第 1.2.2.2 节](../chapter-1-CUDA简介/1.2-编程模型.md#1222-warps-and-simt-线程束和单指令多线程) 描述了 GPU 执行的基本 SIMT 模型，[第 3.2.3 节]() 提供了 SIMT 模型的更多细节。SIMT 模型允许每个线程保持自己的状态和控制流。从功能的角度来看，每个线程都可以执行独立的代码路径。然而，通过注意内核代码尽量减少同一 warp 中的线程采用不同代码路径的情况，可以获得显著的性能提升。

## 2.3.2. 线程层次结构

线程被组织成线程块，线程块又被组织成网格。网格可以是 1 维、2 维或 3 维的，网格的大小可以在内核内部使用内建变量 `gridDim` 查询。线程块也可以是 1 维、2 维或 3 维的。线程块的大小可以在内核内部使用内建变量 `blockDim` 查询。线程块的索引可以使用内建变量 `blockIdx` 查询。在线程块内部，线程的索引使用内建变量 `threadIdx` 获取。这些内建变量用于为每个线程计算唯一的全局线程索引，从而使每个线程能够从全局内存加载/存储特定的数据，并根据需要执行独特的代码路径。

> **C++**
> *   `gridDim.[x|y|z]`：分别在 `x`、`y` 和 `z` 维度上的网格大小。这些值对所有线程都相同，并且是kernel启动时设置的执行配置的一部分。
> *   `blockDim.[x|y|z]`：分别在 `x`、`y` 和 `z` 维度上的块大小。这些值对所有线程都相同，并且是kernel启动时设置的执行配置的一部分。
> *   `blockIdx.[x|y|z]`：分别在 `x`、`y` 和 `z` 维度上的块索引。这些值在不同线程之间会有所不同，用于指示正在执行哪个线程块。
> *   `threadIdx.[x|y|z]`：分别在 `x`、`y` 和 `z` 维度上的线程索引。这些值在不同线程之间会有所不同，用于指示正在执行哪个线程。

> **Python**
> *   `cuda.gridDim.[xyz]`：分别在 `x`、`y` 和 `z` 维度上的网格大小。这些值对所有线程都相同，并且是kernel启动时设置的执行配置的一部分。
> *   `cuda.blockDim.[xyz]`：分别在 `x`、`y` 和 `z` 维度上的块大小。这些值对所有线程都相同，并且是kernel启动时设置的执行配置的一部分。
> *   `cuda.blockIdx.[xyz]`：分别在 `x`、`y` 和 `z` 维度上的块索引。这些值在不同线程之间会有所不同，用于指示正在执行哪个线程块。
> *   `cuda.threadIdx.[xyz]`：分别在 `x`、`y` 和 `z` 维度上的线程索引。这些值在不同线程之间会有所不同，用于指示正在执行哪个线程。


使用多维线程块和网格只是为了方便，不影响性能。一个块的线程以可预测的方式线性化：第一个索引 `x` 移动最快，然后是 `y`，最后是 `z`。这意味着在线程索引的线性化中，`threadIdx.x` 的连续值表示连续的线程，`threadIdx.y` 的步长为 `blockDim.x`，`threadIdx.z` 的步长为 `blockDim.x * blockDim.y`。这会影响线程如何分配给 warp，详见[硬件多线程]()。

### 2.3.3.1. 全局内存

全局内存（也称为设备内存）是用于存储内核中所有线程都可访问的数据的主要内存空间。它类似于 CPU 系统中的 RAM。在 GPU 上运行的内核可以直接访问全局内存，就像在 CPU 上运行的代码可以访问系统内存一样。

全局内存是持久性的。也就是说，全局内存中分配的内存及其存储的数据将一直存在，直到分配被释放或应用程序终止。`cudaDeviceReset` 也会释放所有分配。

全局内存通过 CUDA API 调用（如 `cudaMalloc` 和 `cudaMallocManaged`）进行分配。可以使用 CUDA 运行时 API 调用（如 `cudaMemcpy`）将数据从 CPU 内存复制到全局内存中。使用 CUDA API 分配的全局内存使用 `cudaFree` 释放。

在内核启动之前，全局内存由 CUDA API 调用分配和初始化。在内核执行期间，CUDA 线程可以读取全局内存中的数据，并且 CUDA 线程执行操作的结果可以写回全局内存。一旦内核完成执行，它写入全局内存的结果可以复制回主机或由 GPU 上的其他内核使用。

由于网格中的所有线程都可以访问全局内存，因此必须注意避免线程之间的数据竞争。由于从主机启动的 CUDA 内核的返回类型为 `void`，内核计算出的数值结果返回主机的唯一方法是将这些结果写入全局内存。

下面展示的内核是一个说明全局内存使用的简单示例，其中三个数组 `A`、`B` 和 `C` 位于全局内存中，并由这个向量加法内核访问。

```C++
__global__ void vecAdd(float* A, float* B, float* C, int vectorLength)
{
    int workIndex = threadIdx.x + blockIdx.x*blockDim.x;
    if(workIndex < vectorLength)
    {
        C[workIndex] = A[workIndex] + B[workIndex];
    }
}
```

```python
@cuda.jit
def vecadd(A, B, C):
    work_index = cuda.grid(1)
    C[work_index] = A[work_index] + B[work_index]
```


### 2.3.3.2. 共享内存

共享内存是一个内存空间，可以被一个线程块中的所有线程访问。它物理上位于每个 SM 上，并使用与 L1 缓存（统一数据缓存）相同的物理资源。共享内存中的数据在内核执行期间一直存在。共享内存可以被视为用户管理的暂存器，用于内核执行期间。虽然与全局内存相比大小较小，但由于共享内存位于每个 SM 上，因此其带宽比访问全局内存更高，延迟更低。

由于共享内存可以被一个线程块中的所有线程访问，因此必须注意避免同一线程块中线程之间的数据竞争。同一线程块中的线程之间的同步可以使用 C++ 中的 `__syncthreads()` 函数来实现。在 Python 中，使用 `cuda.syncthreads()` 实现相同的线程块同步。此函数将阻塞线程块中的所有线程，直到所有线程都到达 `__syncthreads()` 或 `cuda.syncthreads()` 调用。

```C++
// assuming blockDim.x is 128
__global__ void example_syncthreads(int* input_data, int* output_data) 
{
    __shared__ int shared_data[128];
    shared_data[threadIdx.x] = input_data[blockDim.x*blockIdx.x + threadIdx.x];

    // All threads synchronize, guaranteeing all writes to 'shared_data' are ordered 
    // before any thread is unblocked from '__syncthreads()':
    __syncthreads();

    // A single thread safely reads 'shared_data':
    if (threadIdx.x == 0) {
        float sum = 0;
        for (int i = 0; i < blockDim.x; ++i) {
            sum += shared_data[i];
        }
        output_data[blockIdx.x] = sum;
    }
}
```

```python
import numpy as np
from numba import cuda
import cupy as cp
```
```python
@cuda.jit
def example_syncthreads(input_data, output_data):
    shared_data = cuda.shared.array(shape=128, dtype=np.int32)
    
    shared_data[cuda.threadIdx.x] = input_data[cuda.blockIdx.x*cuda.blockDim.x + cuda.threadIdx.x]
    cuda.syncthreads()

    if cuda.threadIdx.x == 0:
        sum = 0.0
        for x in shared_data: 
            sum = sum + x
        output_data[cuda.blockIdx.x] = sum
```


共享内存的大小取决于所使用的 GPU 架构。由于共享内存和 L1 缓存共享相同的物理空间，使用共享内存会减少内核可用的 L1 缓存大小。此外，如果内核不使用共享内存，则整个物理空间将用于 L1 缓存。CUDA 运行时 API 提供了函数，使用 `cudaGetDeviceProperties` 函数并查看 `cudaDeviceProp.sharedMemPerMultiprocessor` 和 `cudaDeviceProp.sharedMemPerBlock` 设备属性，来查询每个 SM 和每个线程块的共享内存大小。

CUDA 运行时 API 提供了一个函数 `cudaFuncSetCacheConfig`，用于告诉运行时是分配更多空间给共享内存，还是分配更多空间给 L1 缓存。此函数向运行时指定一个偏好，但不保证会被采纳。运行时可以根据可用资源和内核的需要自由做出决定。

共享内存可以静态分配，也可以动态分配。

#### 2.3.3.2.1. 共享内存的静态分配

要静态分配共享内存，程序员必须在内核内部使用 C++ 中的 `__shared__` 说明符或 Python 中的 `cuda.shared.array()` 声明一个变量。该数组将在共享内存中分配，并将持续内核执行的整个持续时间。以这种方式声明的共享内存的大小必须在编译时指定。例如，以下位于内核体内的代码片段声明了一个类型为 `float`、包含 1024 个元素的共享内存数组。


```C++
__shared__ float sharedArray[1024];
```

```python
from numba import cuda
import numpy as np

shared_array = cuda.shared.array(shape=1024, dtype=np.float32)
```


在此声明之后，线程块中的所有线程都将可以访问此共享内存数组。

#### 2.3.3.2.2. 共享内存的动态分配

要在 C++ 中动态分配共享内存，程序员可以在三括号表示法中将每个线程块所需的共享内存量（以字节为单位）指定为内核启动的第三个（可选）参数，如下所示：`functionName<<<grid, block, sharedMemoryBytes>>>()`。当未指定此参数时，默认为 0。

在 Python 中，必须使用 `cuda.core.launch()` 来启动内核。`LaunchConfig` 参数接受一个 `cuda.core.LaunchConfig` 对象，该对象有一个名为 `shmem_size` 的字段，其功能与 C++ 中三括号表示法的第三个参数相同。

在内核内部，程序员可以使用 C++ 中的 `extern __shared__` 说明符和空的 `[]` 来声明一个将在内核启动时动态分配的变量。在 Python 中，使用与静态分配共享内存相同的 `cuda.shared.array` 方法，并将 `shape` 参数设置为 0。


```C++
extern __shared__ float sharedArray[];
```

```python
from numba import cuda
import numpy as np

## setting shape=0 indicates that this array will be dynamically sized by the kernel launch's execution configuration
shared_array = cuda.shared.array(shape=0, type=np.float32)
```


需要注意的一点是，一个内核只能有一个动态分配的共享数组。如果需要多个动态分配的共享内存数组，必须分配一个足够大的单个动态分配的共享内存数组来包含所有需要的数组，并进行手动分区。例如，在 C++ 中，如果希望在动态分配的共享内存中获得以下等效项：


```C++
short array0[128];
float array1[64];
int   array2[256];
```

可以通过以下方式声明和初始化数组：


```C++
extern __shared__ float array[];

short* array0 = (short*)array;
float* array1 = (float*)&array0[128];
int*   array2 =   (int*)&array1[64];
```


请注意，指针需要与它们指向的类型对齐，因此，例如，以下代码不起作用，因为 `array1` 没有对齐到 4 字节。


```C++
extern __shared__ float array[];
short* array0 = (short*)array;
float* array1 = (float*)&array0[127];
```


在 Python 中，由于没有指针，因此无法使用这种形式的类型双关（type punning）。

### 2.3.3.3. 寄存器

寄存器位于 SM 上，具有线程本地范围。寄存器使用由编译器管理，寄存器在内核执行期间用于线程本地存储。每个 SM 的寄存器数量和每个线程块的寄存器数量可以使用 GPU 的 `regsPerMultiprocessor` 和 `regsPerBlock` 设备属性进行查询。

在编译 C++ 代码时，NVCC 允许开发人员通过 `-maxrregcount` 选项指定内核使用的最大寄存器数量。使用此选项减少内核可以使用的寄存器数量可能会导致更多线程块同时调度在 SM 上，但也可能导致更多的寄存器溢出。寄存器溢出意味着当前存储在片上寄存器中的值必须写出到全局内存，然后再读回，以便为其他值腾出空间。

### 2.3.3.4. 本地内存

本地内存是与寄存器类似的线程本地存储，由 NVCC 管理，但本地内存的物理位置在全局内存空间中。“本地”标签指的是其逻辑范围，而不是其物理位置。本地内存用于内核执行期间的线程本地存储。编译器可能将其放置在本地内存中的自动变量包括：

*   无法确定其使用常量索引的数组，
*   会消耗过多寄存器空间的大型结构体或数组，
*   如果内核使用的寄存器多于可用寄存器，即寄存器溢出时的任何变量。

由于本地内存空间位于设备内存中，因此本地内存访问具有与全局内存访问相同的延迟和带宽，并且受到与[合并全局内存访问](#2341-合并全局内存访问)中描述的相同的内存合并要求。然而，本地内存的组织方式是连续的 32 位字由连续的线程 ID 访问。因此，只要一个 warp 中的所有线程访问相同的相对地址（例如，数组变量中的相同索引或结构体变量中的相同成员），访问就是完全合并的。

### 2.3.3.5. 常量内存

常量内存具有网格范围，并且在应用程序的生命周期内可访问。常量内存位于设备上，对内核来说是只读的。

在 C++ 中，变量或数组使用 `__constant__` 说明符在任何内核或函数外部声明。

在 Python 中，方法 `const_array = numba.cuda.const.array_like(ary)` 在**内核代码内部**指定，以创建一个常量内存数组，其中包含存储在主机数组 `ary` 中的数据。

常量内存意味着一个变量：

*   驻留在常量内存空间中，
*   每个设备有一个不同的对象，
*   在 C++ 中，可以从网格内的所有线程以及通过运行时库从主机访问（`cudaGetSymbolAddress()` / `cudaGetSymbolSize()` / `cudaMemcpyToSymbol()` / `cudaMemcpyFromSymbol()`）。

在 C++ 中，常量内存的生命周期与其创建所在的上下文相同。在 Python 中，常量内存的生命周期与其声明所在的内核相同。

可以使用 `totalConstMem` 设备属性元素查询常量内存的总量。

常量内存对于每个线程将以只读方式使用的小量数据非常有用。与其他内存相比，常量内存较小，通常每个设备为 64KB。

下面是一个声明和使用常量内存的示例片段。


```C++
// In your .cu file
__constant__ float coeffs[4];

__global__ void compute(float *out) {
    int idx = threadIdx.x;
    out[idx] = coeffs[0] * idx + coeffs[1];
}

// In your host code
float h_coeffs[4] = {1.0f, 2.0f, 3.0f, 4.0f};
cudaMemcpyToSymbol(coeffs, h_coeffs, sizeof(h_coeffs));
compute<<<1, 10>>>(device_out);
```

```python
from numba import cuda
import numpy as np

host_array = np.zeros(128, dtype=np.float32)
## fill host_array with other data

@cuda.jit
def kernel(args):
    ...

    const_array = cuda.const.array_like(a)

    # this access now goes through constant memory
    a = const_array[cuda.threadIdx.x]
```


### 2.3.3.6. 缓存

GPU 设备具有多级缓存结构，包括 L2 和 L1 缓存。

L2 缓存位于设备上，并在所有 SM 之间共享。可以使用函数 `cudaGetDeviceProperties` 中的 `l2CacheSize` 设备属性元素查询 L2 缓存的大小。

如上面[共享内存](#2332-共享内存)所述，L1 缓存物理上位于每个 SM 上，并且是与共享内存相同的物理空间。如果内核不使用共享内存，则整个物理空间将用于 L1 缓存。

可以通过允许开发人员指定各种缓存行为的函数来控制 L2 和 L1 缓存。这些函数的详细信息请参见[配置 L1/共享内存平衡]()、[L2 缓存控制]()和[低级加载和存储函数]()。

如果不使用这些提示，编译器和运行时将尽力高效地利用缓存。

### 2.3.3.7. 纹理和表面内存

> **注意**
> 一些较旧的 CUDA 代码可能使用纹理内存，因为在较旧的 NVIDIA GPU 中，这样做在某些情况下可以提供性能优势。在所有当前支持的 GPU 上，这些情况可以使用直接的加载和存储指令来处理，并且对于非纹理加载，使用纹理和表面内存指令不再提供任何性能优势。

GPU 可能具有用于从图像加载数据以用作 3D 渲染中的纹理的专用指令。CUDA 在[纹理对象 API](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__TEXTURE__OBJECT.html) 和[表面对象 API](https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__SURFACE__OBJECT.html) 中公开了这些指令以及使用它们的机制。

对于任何当前支持的 NVIDIA GPU 上的 CUDA 非图形应用程序，纹理和表面内存不提供任何性能优势。当为渲染读取纹理或表面数据时，例如为使用 CUDA 作为其着色器语言的 [NVIDIA OptiX](https://developer.nvidia.com/rtx/ray-tracing/optix) 编写命中着色器时，这些 API 仍然有用。

对于仍在为非纹理加载使用这些 API 的现有代码库的开发人员，可以在传统的 [CUDA C++ 编程指南](https://docs.nvidia.com/cuda/archive/13.0.0/cuda-c-programming-guide/index.html#texture-and-surface-memory)中找到对这些 API 的解释。

### 2.3.3.8. 分布式共享内存

> **注意**
> 分布式共享内存是使用[线程块簇](../chapter-1-CUDA简介/1.2-编程模型.md#12211-thread-block-clusters-线程块簇)时可用的一个功能。线程块簇使用[协作组 API]()，该 API 目前仅在 C++ 中可用。

在计算能力 9.0 中引入并由[协作组]()促进的[线程块簇](../chapter-1-CUDA简介/1.2-编程模型.md#12211-thread-block-clusters-线程块簇)，提供了线程块簇中的线程访问该簇中所有参与线程块的共享内存的能力。这种分区的共享内存称为*分布式共享内存*，相应的地址空间称为分布式共享内存地址空间。属于线程块簇的线程可以在分布式地址空间中读取、写入或执行原子操作，无论该地址是属于本地线程块还是远程线程块。无论内核是否使用分布式共享内存，共享内存大小规格（静态或动态）仍然是每个线程块的。分布式共享内存的大小只是每个簇的线程块数乘以每个线程块的共享内存大小。

访问分布式共享内存中的数据需要所有线程块都存在。用户可以使用来自[类 cluster_group]() 的 `cluster.sync()` 来保证所有线程块都已开始执行。用户还需要确保所有分布式共享内存操作在线程块退出之前完成，例如，如果一个远程线程块试图读取给定线程块的共享内存，程序需要确保远程线程块读取的共享内存在其退出之前已完成。

让我们来看一个简单的直方图计算，以及如何使用线程块簇在 GPU 上进行优化。计算直方图的标准方法是在每个线程块的共享内存中执行计算，然后执行全局内存原子操作。这种方法的一个限制是共享内存容量。一旦直方图存储桶不再适合共享内存，用户就需要直接计算直方图，从而在全局内存中进行原子操作。通过分布式共享内存，CUDA 提供了一个中间步骤，根据直方图存储桶的大小，可以在共享内存、分布式共享内存或直接全局内存中计算直方图。

下面的 CUDA 内核示例展示了如何根据直方图存储桶的数量，在共享内存或分布式共享内存中计算直方图。

```C++
#include <cooperative_groups.h>

// Distributed Shared memory histogram kernel
__global__ void clusterHist_kernel(int *bins, const int nbins, const int bins_per_block, const int *__restrict__ input,
                                    size_t array_size)
{
    extern __shared__ int smem[];
    namespace cg = cooperative_groups;
    int tid = cg::this_grid().thread_rank();

    // Cluster initialization, size and calculating local bin offsets.
    cg::cluster_group cluster = cg::this_cluster();
    unsigned int clusterBlockRank = cluster.block_rank();
    int cluster_size = cluster.dim_blocks().x;

    for (int i = threadIdx.x; i < bins_per_block; i += blockDim.x)
    {
    smem[i] = 0; //Initialize shared memory histogram to zeros
    }

    // cluster synchronization ensures that shared memory is initialized to zero in
    // all thread blocks in the cluster. It also ensures that all thread blocks
    // have started executing and they exist concurrently.
    cluster.sync();

    for (int i = tid; i < array_size; i += blockDim.x * gridDim.x)
    {
    int ldata = input[i];

    //Find the right histogram bin.
    int binid = ldata;
    if (ldata < 0)
        binid = 0;
    else if (ldata >= nbins)
        binid = nbins - 1;

    //Find destination block rank and offset for computing
    //distributed shared memory histogram
    int dst_block_rank = (int)(binid / bins_per_block);
    int dst_offset = binid % bins_per_block;

    //Pointer to target block shared memory
    int *dst_smem = cluster.map_shared_rank(smem, dst_block_rank);

    //Perform atomic update of the histogram bin
    atomicAdd(dst_smem + dst_offset, 1);
    }

    // cluster synchronization is required to ensure all distributed shared
    // memory operations are completed and no thread block exits while
    // other thread blocks are still accessing distributed shared memory
    cluster.sync();

    // Perform global memory histogram, using the local distributed memory histogram
    int *lbins = bins + cluster.block_rank() * bins_per_block;
    for (int i = threadIdx.x; i < bins_per_block; i += blockDim.x)
    {
    atomicAdd(&lbins[i], smem[i]);
    }
}
```

上述内核可以在运行时以簇大小启动，该簇大小取决于所需的分布式共享内存量。如果直方图足够小，可以仅适合一个块的共享内存，用户可以以簇大小 1 启动内核。下面的代码片段展示了如何根据共享内存需求动态启动簇内核。


```C++
// Launch via extensible launch
{
    cudaLaunchConfig_t config = {0};
    config.gridDim = array_size / threads_per_block;
    config.blockDim = threads_per_block;

    // cluster_size depends on the histogram size.
    // ( cluster_size == 1 ) implies no distributed shared memory, just thread block local shared memory
    int cluster_size = 2; // size 2 is an example here
    int nbins_per_block = nbins / cluster_size;

    //dynamic shared memory size is per block.
    //Distributed shared memory size =  cluster_size * nbins_per_block * sizeof(int)
    config.dynamicSmemBytes = nbins_per_block * sizeof(int);

    CUDA_CHECK(::cudaFuncSetAttribute((void *)clusterHist_kernel, cudaFuncAttributeMaxDynamicSharedMemorySize, config.dynamicSmemBytes));

    cudaLaunchAttribute attribute[1];
    attribute[0].id = cudaLaunchAttributeClusterDimension;
    attribute[0].val.clusterDim.x = cluster_size;
    attribute[0].val.clusterDim.y = 1;
    attribute[0].val.clusterDim.z = 1;

    config.numAttrs = 1;
    config.attrs = attribute;

    cudaLaunchKernelEx(&config, clusterHist_kernel, bins, nbins, nbins_per_block, input, array_size);
}
```


## 2.3.4. 内存性能

确保正确的内存使用是在 CUDA 内核中实现高性能的关键。本节讨论在 CUDA 内核中从全局内存和共享内存实现高内存吞吐量的一些一般原则和示例。全局内存性能是大多数内核的首要考虑因素。当线程操作它们未显式加载或创建的数据时，通常会使用共享内存，因此了解共享内存与性能相关的特性也很重要。

以下小节将通过逐步改进示例矩阵转置内核中对全局内存和共享内存的访问，来说明内存访问的重要方面。

### 2.3.4.1. 合并全局内存访问

全局内存通过 32 字节的内存事务进行访问。当一个 CUDA 线程从全局内存请求一个字的数据时，相关的 warp 会将该 warp 中所有线程的内存请求合并为满足请求所需的内存事务数，具体取决于每个线程访问的字的大小以及内存地址在线程之间的分布。例如，如果一个线程请求一个 4 字节的字，warp 将生成的到全局内存的实际内存事务总共将为 32 字节。为了最有效地使用内存系统，warp 应该使用在单个内存事务中获取的所有内存。也就是说，如果一个线程从全局内存请求一个 4 字节的字，并且事务大小为 32 字节，如果该 warp 中的其他线程可以使用来自该 32 字节请求的其他 4 字节数据字，这将导致内存系统的最有效使用。

作为一个简单的例子，如果 warp 中的连续线程请求内存中连续的 4 字节字，那么 warp 将总共请求 128 字节的内存，并且这所需的 128 字节将通过四个 32 字节的内存事务获取。这导致 warp 中的线程对内存事务的利用率为 100%。[图 12](../assets/img/perfect_coalescing_32byte_segments.png) 说明了这个完美合并内存访问的示例。

实现合并内存访问最直接的方法是让连续线程访问内存中的连续元素。例如，对于使用一维线程块启动的内核，之前展示的向量加法内核将实现合并内存访问。注意一个线程如何访问这三个数组：连续线程（由连续的 `workIndex` 值指示）访问数组中的连续元素。

```C++
__global__ void vecAdd(float* A, float* B, float* C, int vectorLength)
{
    int workIndex = threadIdx.x + blockIdx.x*blockDim.x;
    if(workIndex < vectorLength)
    {
        C[workIndex] = A[workIndex] + B[workIndex];
    }
}
```

```python
## Defines a CUDA kernel to perform C = A + B vector addition
@cuda.jit
def vecadd(A, B, C):
    work_index = cuda.grid(1)
    C[work_index] = A[work_index] + B[work_index]
```

实现合并内存访问并不要求连续线程访问连续的内存元素，这只是实现合并的典型方式。只要 warp 中的不同线程以某种线性或置换方式从相同的 32 字节内存段访问元素，就会发生合并内存访问。换句话说，实现合并内存访问的最佳方法是最大化已使用字节与已传输字节的比率。

概念化全局内存合并的另一种等效方法是考虑满足来自单个 warp 的单个加载指令请求的 32 个地址需要多少次全局内存事务。在最好的情况下，只需要一次全局内存事务即可满足所有加载。对于对 4 字节数据元素的完美合并访问，将需要 4 次全局内存事务。在最坏的情况下，可能需要 32 次全局内存事务来满足来自单个 warp 的单个加载指令请求的地址。通常，当满足加载所需的全局内存事务数量尽可能少时，性能最佳。

> **注意**
> 确保全局内存访问的正确合并是编写高性能 CUDA 内核最重要的性能考虑因素之一。应用程序必须尽可能高效地使用内存系统。

#### 2.3.4.1.1. 使用全局内存的矩阵转置示例

```C++
/* macro to index a 1D memory array with 2D indices in row-major order */
/* ld is the leading dimension, i.e. the number of columns in the matrix     */

#define INDX( row, col, ld ) ( ( (row) * (ld) ) + (col) )

/* CUDA kernel for matrix transpose */

__global__ void cuda_transpose(int m, float *a, float *c )
{
    int myCol = blockDim.x * blockIdx.x + threadIdx.x;
    int myRow = blockDim.y * blockIdx.y + threadIdx.y;

    if( myRow < m && myCol < m )
    {
        c[INDX( myCol, myRow, m )] = a[INDX( myRow, myCol, m )];
    } /* end if */
    return;
} /* end cuda_transpose */
```

```python
import numpy as np
from numba import cuda
import cupy as cp

## Matrix transpose kernel, one thread per matrix element launched with 
## 2D thread block on 2D grid to match matrix size
@cuda.jit
def transpose(a, c):
    col = cuda.blockDim.x * cuda.blockIdx.x + cuda.threadIdx.x
    row = cuda.blockDim.y * cuda.blockIdx.y + cuda.threadIdx.y
    c[(col,row)] = a[(row,col)]
```

要确定此内核是否实现了合并内存访问，需要确定连续线程是否正在访问内存中的连续元素。在 2D 线程块中，`x` 索引移动最快，因此 `threadIdx.x` 的连续值应该访问内存中的连续元素。`threadIdx.x` 出现在 `myCol` 中，并且可以观察到，当 `myCol` 是 `INDX` 宏的第二个参数时，连续线程正在读取 `a` 的连续值，因此对 `a` 的读取是完全合并的。

然而，对 `c` 的写入不是合并的，因为 `threadIdx.x` 的连续值（再次检查 `myCol`）正在向 `c` 写入彼此相隔 `ld`（前导维度）个元素的位置。这是因为现在 `myCol` 是 `INDX` 宏的第一个参数，并且当 `myCol` 递增 1 时，内存位置会变化 `ld`。当 `ld` 大于 32 时（只要矩阵大小大于 32 就会发生），这相当于[图 13](../assets/img/no_coalescing_32byte_segments.png)中显示的病态情况。

为了缓解这些未合并的写入，可以使用共享内存，这将在下一节中描述。

### 2.3.4.2. 共享内存访问模式

共享内存有 32 个存储体，其组织方式使得连续的 32 位字映射到连续的存储体。每个存储体的带宽为每个时钟周期 32 位。

当同一 warp 中的多个线程尝试访问同一存储体中的不同元素时，就会发生存储体冲突。在这种情况下，对该存储体中数据的访问将被序列化，直到所有请求该数据的线程都获得该存储体中的数据为止。这种访问的序列化会导致性能损失。

这种情况有两个例外：当同一 warp 中的多个线程访问（读取或写入）相同的共享内存位置时。对于读取访问，该字会被广播到请求的线程。对于写入访问，每个共享内存地址仅由一个线程写入（哪个线程执行写入是未定义的）。

#### 2.3.4.2.1. 使用共享内存的矩阵转置示例

在之前的示例[使用全局内存的矩阵转置示例](#writing-cuda-kernels-matrix-transpose-example-global-memory)中，实现了一个功能正确但未优化全局内存有效使用的矩阵转置，因为对 `c` 矩阵的写入未正确合并。在这个示例中，共享内存将被视为用户管理的缓存，用于暂存来自全局内存的加载和存储，从而实现对读取和写入的合并全局内存访问。

```C++
#define THREADS_PER_BLOCK_X 32
#define THREADS_PER_BLOCK_Y 32

/* macro to index a 1D memory array with 2D indices in column-major order */
/* ld is the leading dimension, i.e. the number of rows in the matrix     */

#define INDX( row, col, ld ) ( ( (col) * (ld) ) + (row) )

/* CUDA kernel for shared memory matrix transpose */
__global__ void smem_transpose(int m,
                                    float *a,
                                    float *c )
{

    /* declare a statically allocated shared memory array */

    __shared__ float smemArray[THREADS_PER_BLOCK_X][THREADS_PER_BLOCK_Y];

    /* determine my row and column indices for the error checking code */

    const int myRow = blockDim.x * blockIdx.x + threadIdx.x;
    const int myCol = blockDim.y * blockIdx.y + threadIdx.y;

    /* determine my row tile and column tile index */

    const int tileX = blockDim.x * blockIdx.x;
    const int tileY = blockDim.y * blockIdx.y;

    if( myRow < m && myCol < m )
    {
        /* read from global memory into shared memory array */
        smemArray[threadIdx.x][threadIdx.y] = a[INDX( tileX + threadIdx.x, tileY + threadIdx.y, m )];
    } /* end if */

    /* synchronize the threads in the thread block */
    __syncthreads();

    if( myRow < m && myCol < m )
    {
        /* write the result from shared memory to global memory */
        c[INDX( tileY + threadIdx.x, tileX + threadIdx.y, m )] = smemArray[threadIdx.y][threadIdx.x];
    } /* end if */
    return;

} 
```

```python
import numpy as np
from numba import cuda
import cupy as cp

## Matrix transpose kernel, one thread per matrix element launched with 
## 2D thread block on 2D grid to match matrix size
## Stages input to shared memory
@cuda.jit
def smem_transpose(a, c):
    smemArray = cuda.shared.array(shape=(32, 32), dtype=np.float32)

    tile_col = cuda.blockDim.x * cuda.blockIdx.x
    tile_row = cuda.blockDim.y * cuda.blockIdx.y

    smemArray[(cuda.threadIdx.x, cuda.threadIdx.y)] = a[(tile_row + cuda.threadIdx.y, tile_col + cuda.threadIdx.x)]

    cuda.syncthreads()
    
    c[(tile_col + cuda.threadIdx.y, tile_row + cuda.threadIdx.x)] = smemArray[(cuda.threadIdx.y, cuda.threadIdx.x)]
```


本例中说明的基本性能优化是确保在访问全局内存时，内存访问被正确合并。在执行复制之前，每个线程计算其 `tileRow` 和 `tileCol` 索引。这些是将被操作的特定块的索引，并且这些块索引基于正在执行的线程块。同一线程块中的每个线程具有相同的 `tileRow` 和 `tileCol` 值，因此可以将其视为此特定线程块将操作的块的起始位置。

然后，内核继续，每个线程块使用以下语句将矩阵的一个 32 x 32 块从全局内存复制到共享内存。由于 warp 的大小为 32 个线程，此复制操作将由 32 个 warp 执行，warp 之间没有顺序保证。


```C++
smemArray[threadIdx.x][threadIdx.y] = a[INDX( tileRow + threadIdx.y, tileCol + threadIdx.x, m )];
```

```python
smemArray[(cuda.threadIdx.x, cuda.threadIdx.y)] = a[(tile_row + cuda.threadIdx.y, tile_col + cuda.threadIdx.x)]
```


注意，由于 `threadIdx.x` 出现在 `INDX` 的第二个参数中，连续线程正在访问内存中的连续元素，因此对 `a` 的读取是完全合并的。在 Python 中，使用 `cuda.threadIdx.x` 作为元组中的最后一个索引具有相同的效果：对 `a` 的访问是完全合并的。

内核的下一步是调用 `__syncthreads()` / `cuda.syncthreads()`。这确保线程块中的所有线程在继续之前都已执行完之前的代码，从而确保在对 `a` 的写入完成到共享内存之后才进行下一步。这一点至关重要，因为下一步将涉及线程从共享内存读取。如果没有 `__syncthreads()`/ `cuda.syncthreads()` 调用，就不能保证线程块中的所有 warp 在部分 warp 前进到从共享内存中的数组读取之前，都完成了将 `a` 读入共享内存的操作。当线程处理或存储它未加载的数据时，需要同步以确保在访问该元素之前该元素的加载操作已完成。

在内核的此时，对于每个线程块，共享内存数组拥有矩阵的一个 32 x 32 的块，其排列顺序与原始矩阵相同。为了确保块内的元素被正确转置，当从 `smemArray` 读取时，`threadIdx.x` 和 `threadIdx.y` 被交换。为了确保整个块被放置在 `c` 中的正确位置，当写入 `c` 时，`tileRow` 和 `tileCol` 索引也被交换。为了确保正确的合并，`threadIdx.x` 被用作 `INDX` 的第二个参数，如下面的语句所示。在 Python 中，这再次通过使用 `cuda.threadIdx.x` 作为矩阵索引元组的最后一个索引来实现。


```C++
c[INDX( tileCol + threadIdx.y, tileRow + threadIdx.x, m )] = smemArray[threadIdx.y][threadIdx.x];
```

```python
c[(tile_col + cuda.threadIdx.y, tile_row + cuda.threadIdx.x)] = smemArray[(cuda.threadIdx.y, cuda.threadIdx.x)]
```


该内核展示了共享内存的两种常见用途。

*   共享内存用于暂存来自全局内存的数据，以确保对全局内存的读取和写入都正确合并。
*   共享内存用于允许同一线程块中的线程之间共享数据。

#### 2.3.4.2.2. 共享内存存储体冲突

在[第 2.3.4.2 节](#writing-cuda-kernels-shared-memory-access-patterns)中，描述了共享内存的存储体结构。在之前的矩阵转置示例中，实现了与/从全局内存的正确合并内存访问，但没有考虑是否存在共享内存存储体冲突。考虑以下 2D 共享内存声明：


```C++
__shared__ float smemArray[32][32];
```

```python
from numba import cuda
import numpy as np

smemArray = cuda.shared.array(shape=(32,32), dtype=np.float32)
```


假设内核期望以 32 x 32 线程的 2D 线程块启动。由于一个 warp 是 32 个线程，特定 warp 中的每个线程将具有固定的 `threadIdx.y` 值，并且具有 `0 <= threadIdx.x < 32`。

[图 17](../assets/img/bank-conflicts-shared-mem.png) 的左图说明了当 warp 中的线程访问 `smemArray` 的一列数据时的情况。Warp 0 正在访问从 `smemArray[0][0]` 到 `smemArray[31][0]`（在 Python 中为 `smemArray[(0,0)]` 到 `smemArray[(31,0)]`）的内存位置。在 C++ 和 Python 中，多维数组的排序都是最后一个索引移动最快，因此 warp 0 中的连续线程正在访问相隔 32 个元素的内存位置。如图所示，颜色表示存储体，warp 0 沿着整列的这种访问导致了 32 路存储体冲突。

[图 17](../assets/img/bank-conflicts-shared-mem.png) 的右图说明了当 warp 中的线程访问 `smemArray` 的一行数据时的情况。Warp 0 正在访问从 `smemArray[0][0]` 到 `smemArray[0][31]`（在 Python 中为 `smemArray[(0,0)]` 到 `smemArray[(0,31)]`）的内存位置。在这种情况下，warp 0 中的连续线程正在访问相邻的内存位置。如图所示，颜色表示存储体，warp 0 沿着整行的这种访问导致没有存储体冲突。理想的情况是 warp 中的每个线程都访问不同颜色的共享内存位置。

```C++
smemArray[threadIdx.x][threadIdx.y] = a[INDX( tileRow + threadIdx.y, tileCol + threadIdx.x, m )];
```

```python
smemArray[(cuda.threadIdx.x, cuda.threadIdx.y)] = a[(tile_row + cuda.threadIdx.y, tile_col + cuda.threadIdx.x)]
```

由于数组以行优先顺序存储，同一 warp 中的连续线程（由 `threadIdx.x` 的连续值指示）将访问 `smemArray`，步长为 32 个元素，因为 `threadIdx.x` 是数组的第一个索引。这导致了 32 路存储体冲突，并由[图 17](#writing-cuda-kernels-figure-bank-conflicts-shared-mem) 的左图说明。

共享内存的第二次使用是在将数据从共享内存写回全局内存时：


```C++
c[INDX( tileCol + threadIdx.y, tileRow + threadIdx.x, m )] = smemArray[threadIdx.y][threadIdx.x];
```

```python
c[(tile_col + cuda.threadIdx.y, tile_row + cuda.threadIdx.x)] = smemArray[(cuda.threadIdx.y, cuda.threadIdx.x)]
```


在这种情况下，因为 `threadIdx.x` 是 `smemArray` 的第二个索引，同一 warp 中的连续线程将访问 `smemArray`，步长为 1 个元素。这导致没有存储体冲突，并由[图 17](#writing-cuda-kernels-figure-bank-conflicts-shared-mem) 的右图说明。

如[第 2.3.4.2.1 节](#writing-cuda-kernels-matrix-transpose-example-shared-memory)所示，矩阵转置内核有一次共享内存访问没有存储体冲突，有一次访问有 32 路存储体冲突。避免存储体冲突的一个常见解决方法是通过向数组的列维度添加 1 来填充共享内存，如下所示：


```C++
__shared__ float smemArray[THREADS_PER_BLOCK_X][THREADS_PER_BLOCK_Y+1];
```

```python
smemArray = cuda.shared.array(shape=(32, 32 + 1), dtype=np.float32)
```


对 `smemArray` 声明的这个微小调整将消除存储体冲突。为了说明这一点，请考虑[图 18](../assets/img/no-bank-conflicts-shared-mem.png)，其中共享内存数组已声明为大小 32 x 33。可以看出，无论同一 warp 中的线程是沿着整列还是整行访问共享内存数组，存储体冲突都已消除，即同一 warp 中的线程访问具有不同颜色的位置。

## 2.3.5. 原子操作

高性能 CUDA 内核依赖于尽可能多地表达算法的并行性。GPU 内核执行的异步特性要求线程尽可能独立地操作。线程不可能总是完全独立，正如我们在[共享内存](#2332-共享内存)中看到的，存在一种机制允许同一线程块中的线程交换数据和同步。

在整个网格的级别上，没有这样的机制来同步网格中的所有线程。然而，存在一种通过使用原子函数来提供对全局内存位置的同步访问的机制。原子函数允许线程获取全局内存位置上的锁，并对该位置执行读-修改-写操作。当锁被持有时，其他线程无法访问同一位置。

### 2.3.5.1. C++ std::atomic 风格的原子操作

在 C++ 中，CUDA 提供了与同名的 C++ 标准库原子操作 `cuda::std::atomic` 和 `cuda::std::atomic_ref` 类似的语法和行为。CUDA 还提供了扩展的 C++ 原子操作 `cuda::atomic` 和 `cuda::atomic_ref`，允许用户指定原子操作的[线程范围]()。原子函数的详细信息在[原子函数]()中介绍。

下面是一个使用 `cuda::atomic_ref` 执行设备范围原子加法的示例，其中 `array` 是一个浮点数数组，`result` 是指向全局内存中某个位置的浮点指针，该位置是数组总和将被存储的位置。

```C++
__global__ void sumReduction(int n, float *array, float *result) {
...
tid = threadIdx.x + blockIdx.x * blockDim.x;

cuda::atomic_ref<float, cuda::thread_scope_device> result_ref(result);
result_ref.fetch_add(array[tid]);
...
}
```

应谨慎使用原子函数，因为它们会强制进行线程同步，从而可能影响性能。

### 2.3.5.2. Python 中的内存原子操作

在 Python 中，原子内存操作由 `numba.cuda.atomic` 命名空间中的 GPU 代码可用的函数提供。典型的操作如 `add`、`sub`、`max`、`min` 和 `compare_and_swap` 都是可用的。支持的原子操作的完整列表可在 Numba CUDA 文档中找到。

以下代码显示了一个使用原子内存访问来计算数组所有值之和的内核示例。每个线程块将数组的一个片段加载到共享内存中。每个线程块的单个线程计算局部和，并对结果数组 `s` 执行原子加法。由于数据位于共享内存中，靠近 SM 的计算资源，让单个线程执行求和通常仍然具有良好的性能。

```python
import numpy as np
from numba import cuda
import cupy as cp


@cuda.jit
def sum_reduce(a, s):
    ## create a shared array to support a block size up to 512 threads
    ## even though we'll use fewer in this example
    shared_staging = cuda.shared.array(shape=512, dtype=np.float32)

    ## Load values into shared memory and then synchronize to make sure all loads completed
    shared_staging[cuda.threadIdx.x] = a[cuda.blockIdx.x*cuda.blockDim.x + cuda.threadIdx.x]
    cuda.syncthreads()

    ## only thread 0 of each block does the local additions, followed by a single
    ## atomic operation per thread block
    local_sum = float(0.0)
    if cuda.threadIdx.x == 0:
        for i in range(cuda.blockDim.x):
            local_sum = local_sum + shared_staging[i]
        cuda.atomic.add(s, 0, local_sum)
    

array_length = 2**18

a = cp.ones(array_length)
s = cp.zeros(1, dtype=np.float32)

block_size = 256
grid_size = int(array_length/block_size)
sum_reduce[grid_size, block_size](a, s)

s_host = cp.asnumpy(s)
print(f"Sum is {int(s_host[0])}, expected {array_length}")
```

在这个简单的例子中，输入数组全为 1，因此正确的总和与 `array_length` 相同。

如果代码行

```c++
cuda.atomic.add(s, 0, local_sum)
```

被替换为非原子加法

```c++
s[0] = s[0] + local_sum
```

对 `s[0]` 的访问将不是原子的，`s[0]` 的最终值将小于 `array_length`。此外，该值可能因运行而异，并在具有不同数量 SM 的 GPU 上运行时发生变化。这说明了为什么在此代码中需要原子内存访问才能保证正确性。

> **注意**
> 这个例子虽然功能正确，但目的不是说明如何在 GPU 上编写峰值性能的规约操作。NVIDIA 的 CUDA 核心计算库 (CCCL) 为许多操作（包括规约）提供了高性能原语。为了提高生产效率和性能，开发人员应尽可能优先使用这些高度优化的实现，而不是重新实现相同的算法。这些原语在 Python 中可通过 `cuda.coop` 包获得。

C++ 中也提供了类似的原子操作，在[第 5.4.5.1 节](../05-appendices/cpp-language-extensions.html#legacy-atomic-functions)中讨论，但建议使用 `std::atomic` 风格的原子操作，并认为这是 CUDA C++ 中的最佳实践。

## 2.3.6. 协作组

[协作组]() 是 CUDA C++ 中提供的一个软件工具，允许应用程序定义可以相互同步的线程组，即使该线程组跨越多个线程块、单个 GPU 上的多个网格，甚至跨越多个 GPU。一般来说，CUDA 编程模型允许线程块或线程块簇内的线程高效同步，但不提供指定小于线程块或簇的线程组的机制。同样，CUDA 编程模型不提供允许跨线程块同步的机制或保证。

协作组通过软件提供这两种能力。协作组允许应用程序创建跨越线程块和簇边界的线程组，尽管这样做会带来一些语义限制和性能影响，这在[专门介绍协作组的功能部分]()中有详细描述。

## 2.3.7. 内核启动与占用率

当启动 CUDA 内核时，CUDA 线程根据内核启动时指定的执行配置被分组到线程块和网格中。内核启动后，调度器将线程块分配给 SM。哪些线程块被调度到哪些 SM 上执行的细节无法由应用程序控制或查询，调度器也不做任何顺序保证，因此程序不能依赖特定的调度顺序或方案来保证正确执行。

可以调度在一个 SM 上的块数取决于给定线程块所需的硬件资源以及 SM 上可用的硬件资源。当内核首次启动时，调度器开始将线程块分配给 SM。只要 SM 有未被其他线程块占用的足够硬件资源，调度器就会继续将线程块分配给 SM。如果在某个时刻没有 SM 有能力接受另一个线程块，调度器将等待 SM 完成先前分配的线程块。一旦发生这种情况，SM 就可以自由接受更多工作，调度器将线程块分配给它们。此过程持续到所有线程块都已调度并执行完毕。

`cudaGetDeviceProperties` 函数允许应用程序通过[设备属性](https://docs.nvidia.com/cuda/cuda-runtime-api/structcudaDeviceProp.html#structcudaDeviceProp)查询每个 SM 的限制。请注意，每个 SM 和每个线程块都有其限制。

*   `maxBlocksPerMultiProcessor`：每个 SM 的最大常驻块数。
*   `sharedMemPerMultiprocessor`：每个 SM 可用的共享内存量（以字节为单位）。
*   `regsPerMultiprocessor`：每个 SM 可用的 32 位寄存器数量。
*   `maxThreadsPerMultiProcessor`：每个 SM 的最大常驻线程数。
*   `sharedMemPerBlock`：一个线程块可以分配的最大共享内存量（以字节为单位）。
*   `regsPerBlock`：一个线程块可以分配的最大 32 位寄存器数量。
*   `maxThreadsPerBlock`：每个线程块的最大线程数。

CUDA 内核的占用率是活动 warp 数与 SM 支持的最大活动 warp 数之比。通常，尽可能保持高占用率是一种好的做法，这可以隐藏延迟并提高性能。

要计算占用率，需要知道刚才描述的 SM 资源限制，并且需要知道所讨论的 CUDA 内核需要哪些资源。为了确定每个内核的资源使用情况，在程序编译期间可以使用 `nvcc` 的 `--resource-usage` [选项](https://docs.nvidia.com/cuda/cuda-compiler-driver-nvcc/index.html#resource-usage-res-usage)，它将显示内核所需的寄存器和共享内存数量。

如果一个内核启动为 `testKernel<<<512, 768>>>()`，即每块 768 个线程，每个 SM 一次只能执行 2 个线程块。调度器不能为每个 SM 分配超过 2 个线程块，因为 `maxThreadsPerMultiProcessor` 是 2048。所以占用率将是 (768 * 2) / 2048，即 75%。

如果一个内核启动为 `testKernel<<<512, 32>>>()`，即每块 32 个线程，每个 SM 不会遇到 `maxThreadsPerMultiProcessor` 的限制，但由于 `maxBlocksPerMultiProcessor` 是 32，调度器只能为每个 SM 分配 32 个线程块。由于块中的线程数是 32，SM 上常驻的线程总数将是 32 块 * 每块 32 线程 = 1024 个线程。由于计算能力 10.0 的 SM 每个 SM 的最大常驻线程数为 2048，因此这种情况下的占用率为 1024 / 2048，即 50%。

同样的分析也可以用于共享内存。例如，如果一个内核使用 100KB 的共享内存，调度器只能为每个 SM 分配 2 个线程块，因为该 SM 上的第三个线程块需要另外 100KB 的共享内存，总计 300KB，超过了每个 SM 可用的 233472 字节。

每块线程数和每块共享内存使用量由程序员显式控制，可以调整以达到所需的占用率。程序员对寄存器使用的控制有限，因为编译器和运行时将尝试优化寄存器使用。但是，程序员可以通过 `nvcc` 的 `--maxrregcount` [选项](https://docs.nvidia.com/cuda/cuda-compiler-driver-nvcc/index.html#maxrregcount-amount-maxrregcount)指定每个线程块的最大寄存器数量。如果内核需要的寄存器超过此指定数量，内核很可能会溢出到本地内存，这将改变内核的性能特征。在某些情况下，即使发生溢出，限制寄存器允许调度更多线程块，从而增加占用率，并可能导致性能净提升。
