# 2.1. CUDA C++ 入门

本章通过说明 CUDA 编程模型中的基本概念如何在 C++ 中体现，来介绍这些概念。

本编程指南主要关注 CUDA 运行时 API。CUDA 运行时 API 是在 C++ 中使用 CUDA 最常用的方式，它构建在较低层的 CUDA 驱动 API 之上。

[CUDA 运行时 API 和 CUDA 驱动 API](../chapter-1-CUDA简介/1.3-CUDA平台.md#1321-cuda-runtiom-api-and-cuda-driver-api-cuda运行时api和cuda驱动api) 讨论了这些 API 之间的区别，而 [CUDA 驱动 API]() 则讨论了编写混合使用这些 API 的代码。

本指南假定已安装 CUDA 工具包和 NVIDIA 驱动程序，并且存在受支持的 NVIDIA GPU。关于安装必要 CUDA 组件的说明，请参阅 [CUDA 快速入门指南](https://docs.nvidia.com/cuda/cuda-quick-start-guide/index.html)。

## 2.1.1. 使用 NVCC 编译

用 C++ 编写的 GPU 代码使用 NVIDIA CUDA 编译器 `nvcc` 进行编译。`nvcc` 是一个编译器驱动，它简化了编译 C++ 或 PTX 代码的过程：它提供简单且熟悉的命令行选项，并通过调用实现不同编译阶段的工具集合来执行这些选项。

本指南将展示可在任何安装了 CUDA 工具包的 Linux 系统上、在 Windows 命令提示符或 PowerShell 中，或在安装了 CUDA 工具包的 Windows 子系统 Linux (WSL) 上使用的 `nvcc` 命令行。本指南的 [nvcc 章节]() 涵盖了 `nvcc` 的常见用例，完整的文档由 [nvcc 用户手册](https://docs.nvidia.com/cuda/cuda-compiler-driver-nvcc/index.html) 提供。

## 2.1.2. 内核 (Kernels)

如 [CUDA 编程模型](../chapter-1-CUDA简介/1.2-编程模型.md) 引言中所述，在 GPU 上执行并可从主机调用的函数称为内核。内核的编写旨在由许多并行线程同时执行。

### 2.1.2.1. 指定内核

内核的代码使用 `__global__` 声明说明符来指定。这向编译器表明，该函数将以允许从内核启动中调用的方式为 GPU 编译。内核启动是一个启动内核运行的操作，通常从 CPU 发起。内核是返回类型为 `void` 的函数。

```cuda
// Kernel definition
__global__ void vecAdd(float* A, float* B, float* C)
{

}
```

### 2.1.2.2. 启动内核

将并行执行内核的线程数作为内核启动的一部分来指定。这被称为执行配置。同一内核的不同调用可以使用不同的执行配置，例如不同的线程数或线程块数。

从 CPU 代码启动内核有两种方式：[三括号表示法](#21221-三括号表示法) 和 `cudaLaunchKernelEx`。三括号表示法是最常见的启动内核的方式，这里将对其进行介绍。使用 `cudaLaunchKernelEx` 启动内核的示例将在 [第 3.1.1 节]() 中详细展示和讨论。

#### 2.1.2.2.1. 三括号表示法

三括号表示法是一种 [CUDA C++ 语言扩展]()，用于启动内核。之所以称为三括号，是因为它使用三个尖括号字符来封装内核启动的执行配置，即 `<<< >>>`。执行配置参数以逗号分隔的列表形式指定在尖括号内，类似于函数调用的参数。下面展示了启动 `vecAdd` 内核的语法。

```cuda
 __global__ void vecAdd(float* A, float* B, float* C)
 {
 }

int main()
{
    ...
    // Kernel invocation
    vecAdd<<<1, 256>>>(A, B, C);
    ...
}
```

三括号表示法的前两个参数分别是网格维度和线程块维度。当使用一维线程块或网格时，可以使用整数来指定维度。

上述代码启动了一个包含 256 个线程的单个线程块。每个线程将执行完全相同的内核代码。在 [线程和网格索引内建变量](#2123-线程和网格索引内建变量) 中，我们将展示每个线程如何使用它在线程块和网格中的索引来改变其操作的数据。

每个线程块的线程数有限制，因为一个块的所有线程都位于同一个流式多处理器 (SM) 上，并且必须共享 SM 的资源。在当前 GPU 上，一个线程块最多可包含 1024 个线程。如果资源允许，多个线程块可以同时调度在一个 SM 上。

内核启动相对于主机线程是异步的。也就是说，内核将被设置为在 GPU 上执行，但主机代码不会等待内核在 GPU 上完成（甚至开始）执行就继续执行。必须使用某种形式的 GPU 和 CPU 同步来确定内核已完成。最基本的方式，即完全同步整个 GPU，在 [同步 CPU 和 GPU](#214-同步-cpu-和-gpu) 中展示。更复杂的同步方法在 [异步执行](./2.5-异步执行.md) 中介绍。

当使用二维或三维网格或线程块时，使用 CUDA 类型 `dim3` 作为网格和线程块维度参数。下面的代码片段展示了使用 16×16 的线程块网格启动 `MatAdd` 内核，每个线程块为 8×8。

```cuda
int main()
{
    ...
    dim3 grid(16,16);
    dim3 block(8,8);
    MatAdd<<<grid, block>>>(A, B, C);
    ...
}
```

### 2.1.2.3. 线程和网格索引内建变量

在内核代码中，CUDA 提供了内建变量来访问执行配置的参数以及线程或块的索引。

> *   `threadIdx` 给出线程在其线程块内的索引。一个线程块中的每个线程将有不同的索引。
> *   `blockDim` 给出线程块的维度，该维度在内核启动的执行配置中指定。
> *   `blockIdx` 给出线程块在网格中的索引。每个线程块将有不同的索引。
> *   `gridDim` 给出网格的维度，该维度在内核启动时的执行配置中指定。

这些内建变量中的每一个都是一个具有 `.x`、`.y` 和 `.z` 成员的三分量向量。未在启动配置中指定的维度默认为 1。
`threadIdx` 和 `blockIdx` 是从零开始的索引。也就是说，`threadIdx.x` 的取值范围是从 0 到 `blockDim.x-1`（包括）。`.y` 和 `.z` 在其各自的维度上操作方式相同。

类似地，`blockIdx.x` 的取值范围是从 0 到 `gridDim.x-1`（包括），`.y` 和 `.z` 维度也是如此。

这些允许单个线程识别它应该执行哪些工作。回到 `vecAdd` 内核，该内核接受三个参数，每个参数都是一个浮点数向量。该内核执行 `A` 和 `B` 的元素级加法，并将结果存储在 `C` 中。内核被并行化，使得每个线程执行一次加法。它计算哪个元素由其线程和网格索引决定。

```cuda
__global__ void vecAdd(float* A, float* B, float* C)
{
   // calculate which element this thread is responsible for computing
   int workIndex = threadIdx.x + blockDim.x * blockIdx.x

   // Perform computation
   C[workIndex] = A[workIndex] + B[workIndex];
}

int main()
{
    ...
    // A, B, and C are vectors of 1024 elements
    vecAdd<<<4, 256>>>(A, B, C);
    ...
}
```

在此示例中，使用 4 个线程块，每个块 256 个线程，来添加一个包含 1024 个元素的向量。在第一个线程块中，`blockIdx.x` 为零，因此每个线程的 `workIndex` 就是它的 `threadIdx.x`。在第二个线程块中，`blockIdx.x` 为 1，因此 `blockDim.x * blockIdx.x` 等于 `blockDim.x`，在此例中为 256。第二个线程块中每个线程的 `workIndex` 将是其 `threadIdx.x + 256`。在第三个线程块中，`workIndex` 将是 `threadIdx.x + 512`。

这种 `workIndex` 的计算方式在一维并行化中非常常见。扩展到二维或三维通常在每个维度上都遵循相同的模式。

#### 2.1.2.3.1. 边界检查

上面给出的例子假设向量的长度是线程块大小（本例中为 256 的倍数）的倍数。为了使内核能处理任意向量长度，我们可以添加检查，确保内存访问不超过数组边界，如下所示，然后启动一个线程块，其中将有一些非活动线程。

```cuda
__global__ void vecAdd(float* A, float* B, float* C, int vectorLength)
{
     // calculate which element this thread is responsible for computing
     int workIndex = threadIdx.x + blockDim.x * blockIdx.x

     if(workIndex < vectorLength)
     {
         // Perform computation
         C[workIndex] = A[workIndex] + B[workIndex];
     }
}
```

使用上面的内核代码，可以启动比所需更多的线程，而不会导致对数组的越界访问。当 `workIndex` 超过 `vectorLength` 时，线程退出且不执行任何工作。启动块中不做任何工作的额外线程不会产生很大的开销成本，但应避免启动没有线程执行工作的线程块。此内核现在可以处理不是块大小倍数的向量长度。

所需的线程块数量可以计算为所需线程数（此处为向量长度）除以每个块的线程数，然后向上取整。即所需线程数除以每块线程数的整数除法，向上取整。下面给出了一种将其表示为单个整数除法的常见方法。通过在整数除法之前加上 `threads - 1`，这就像向上取整函数一样，只有当向量长度不能被每块线程数整除时，才添加另一个线程块。

```cuda
// vectorLength is an integer storing number of elements in the vector
int threads = 256;
int blocks = (vectorLength + threads-1)/threads;
vecAdd<<<blocks, threads>>>(devA, devB, devC, vectorLength);
```

CUDA 核心计算库 (CCCL) 提供了一个便捷的工具 `cuda::ceil_div` 来执行此向上取整除法，以计算内核启动所需的块数。通过包含头文件 `<cuda/cmath>` 即可使用此工具。

```cuda
// vectorLength is an integer storing number of elements in the vector
int threads = 256;
int blocks = cuda::ceil_div(vectorLength, threads);
vecAdd<<<blocks, threads>>>(devA, devB, devC, vectorLength);
```

这里选择每块 256 个线程是任意的，但这通常是一个不错的起始值。

## 2.1.3. GPU 计算中的内存

为了使用上面展示的 `vecAdd` 内核，数组 `A`、`B` 和 `C` 必须位于 GPU 可访问的内存中。有几种不同的方法可以做到这一点，这里将说明其中两种。其他方法将在后面关于 [统一内存]() 的章节中介绍。GPU 上运行的代码可用的内存空间在 [GPU 内存](../chapter-1-CUDA简介/1.2-编程模型.md#123-gpu-memory-gpu内存) 中介绍，并在 [GPU 设备内存空间]() 中有更详细的介绍。

### 2.1.3.1. 统一内存

统一内存是 CUDA 运行时的一个特性，它让 NVIDIA 驱动程序管理主机和设备之间的数据移动。内存使用 `cudaMallocManaged` API 或通过使用 `__managed__` 说明符声明变量来分配。NVIDIA 驱动程序将确保每当 GPU 或 CPU 尝试访问该内存时，它都能被访问。

下面的代码展示了一个完整的函数，用于启动 `vecAdd` 内核，该内核使用统一内存来管理将在 GPU 上使用的输入和输出向量。`cudaMallocManaged` 分配可以从 CPU 或 GPU 访问的缓冲区。这些缓冲区使用 `cudaFree` 释放。

```cuda
void unifiedMemExample(int vectorLength)
{
    // Pointers to memory vectors
    float* A = nullptr;
    float* B = nullptr;
    float* C = nullptr;
    float* comparisonResult = (float*)malloc(vectorLength*sizeof(float));

    // Use unified memory to allocate buffers
    cudaMallocManaged(&A, vectorLength*sizeof(float));
    cudaMallocManaged(&B, vectorLength*sizeof(float));
    cudaMallocManaged(&C, vectorLength*sizeof(float));

    // Initialize vectors on the host
    initArray(A, vectorLength);
    initArray(B, vectorLength);

    // Launch the kernel. Unified memory will make sure A, B, and C are
    // accessible to the GPU
    int threads = 256;
    int blocks = cuda::ceil_div(vectorLength, threads);
    vecAdd<<<blocks, threads>>>(A, B, C, vectorLength);
    // Wait for the kernel to complete execution
    cudaDeviceSynchronize();

    // Perform computation serially on CPU for comparison
    serialVecAdd(A, B, comparisonResult, vectorLength);

    // Confirm that CPU and GPU got the same answer
    if(vectorApproximatelyEqual(C, comparisonResult, vectorLength))
    {
        printf("Unified Memory: CPU and GPU answers match\n");
    }
    else
    {
        printf("Unified Memory: Error - CPU and GPU answers do not match\n");
    }

    // Clean Up
    cudaFree(A);
    cudaFree(B);
    cudaFree(C);
    free(comparisonResult);

}
```

统一内存支持所有 CUDA 支持的操作系统和 GPU，但其底层机制和性能可能因系统架构而异。[统一内存]() 提供了更多细节。在某些 Linux 系统上（例如，那些具有 [地址转换服务]() 或 [异构内存管理]() 的系统），所有系统内存都自动是统一内存，无需使用 `cudaMallocManaged` 或 `__managed__` 说明符。

### 2.1.3.2. 显式内存管理

显式管理内存分配和内存空间之间的数据迁移有助于提高应用程序性能，尽管这会使代码更冗长。下面的代码使用 `cudaMalloc` 在 GPU 上显式分配内存。GPU 上的内存使用与上一示例中用于统一内存相同的 `cudaFree` API 来释放。

```cuda
void explicitMemExample(int vectorLength)
{
    // Pointers for host memory
    float* A = nullptr;
    float* B = nullptr;
    float* C = nullptr;
    float* comparisonResult = (float*)malloc(vectorLength*sizeof(float));
    
    // Pointers for device memory
    float* devA = nullptr;
    float* devB = nullptr;
    float* devC = nullptr;

    //Allocate Host Memory using cudaMallocHost API. This is best practice
    // when buffers will be used for copies between CPU and GPU memory
    cudaMallocHost(&A, vectorLength*sizeof(float));
    cudaMallocHost(&B, vectorLength*sizeof(float));
    cudaMallocHost(&C, vectorLength*sizeof(float));

    // Initialize vectors on the host
    initArray(A, vectorLength);
    initArray(B, vectorLength);

    // start-allocate-and-copy
    // Allocate memory on the GPU
    cudaMalloc(&devA, vectorLength*sizeof(float));
    cudaMalloc(&devB, vectorLength*sizeof(float));
    cudaMalloc(&devC, vectorLength*sizeof(float));

    // Copy data to the GPU
    cudaMemcpy(devA, A, vectorLength*sizeof(float), cudaMemcpyDefault);
    cudaMemcpy(devB, B, vectorLength*sizeof(float), cudaMemcpyDefault);
    cudaMemset(devC, 0, vectorLength*sizeof(float));
    // end-allocate-and-copy

    // Launch the kernel
    int threads = 256;
    int blocks = cuda::ceil_div(vectorLength, threads);
    vecAdd<<<blocks, threads>>>(devA, devB, devC, vectorLength);
    // wait for kernel execution to complete
    cudaDeviceSynchronize();

    // Copy results back to host
    cudaMemcpy(C, devC, vectorLength*sizeof(float), cudaMemcpyDefault);

    // Perform computation serially on CPU for comparison
    serialVecAdd(A, B, comparisonResult, vectorLength);

    // Confirm that CPU and GPU got the same answer
    if(vectorApproximatelyEqual(C, comparisonResult, vectorLength))
    {
        printf("Explicit Memory: CPU and GPU answers match\n");
    }
    else
    {
        printf("Explicit Memory: Error - CPU and GPU answers to not match\n");
    }

    // clean up
    cudaFree(devA);
    cudaFree(devB);
    cudaFree(devC);
    cudaFreeHost(A);
    cudaFreeHost(B);
    cudaFreeHost(C);
    free(comparisonResult);
}
```

CUDA API `cudaMemcpy` 用于将数据从 CPU 上的缓冲区复制到 GPU 上的缓冲区。除了目标指针、源指针和字节大小外，`cudaMemcpy` 的最后一个参数是 `cudaMemcpyKind_t`。它可以有以下值：

*   `cudaMemcpyHostToDevice`：用于从 CPU 到 GPU 的复制
*   `cudaMemcpyDeviceToHost`：用于从 GPU 到 CPU 的复制
*   `cudaMemcpyDeviceToDevice`：用于 GPU 内部或 GPU 之间的复制

在此示例中，`cudaMemcpyDefault` 作为最后一个参数传递给 `cudaMemcpy`。这使 CUDA 根据源指针和目标指针的值来确定要执行的复制类型。

`cudaMemcpy` API 是同步的。也就是说，它在复制完成之前不会返回。异步复制在 [在 CUDA 流中启动内存传输](./2.5-异步执行.md#2523-在-cuda-流中启动内存传输) 中介绍。

该代码使用 `cudaMallocHost` 在 CPU 上分配内存。这分配了主机上的 [页锁定内存]()，可以提高复制性能，并且对于 [异步](./2.5-异步执行.md) 内存传输是必需的。通常，对于将用于与 GPU 进行数据传输的 CPU 缓冲区，使用页锁定内存是一种好习惯。如果系统上页锁定的主机内存过多，性能可能会下降。最佳实践是仅页锁定将用于向 GPU 发送数据或从 GPU 接收数据的缓冲区。

### 2.1.3.3. 内存管理与应用程序性能

从上面的例子可以看出，显式内存管理更冗长，要求程序员指定主机和设备之间的复制。这是显式内存管理的优点和缺点：它提供了对数据何时在主机和设备之间复制、内存驻留位置以及确切内存分配位置的更多控制。显式内存管理可以通过控制内存传输并将其与其他计算重叠来提供性能提升的机会。

使用统一内存时，有一些 CUDA API（将在 [内存建议和预取]() 中介绍），它们为管理内存的 NVIDIA 驱动程序提供提示，这使得在使用统一内存时能够获得使用显式内存管理的一些性能优势。

## 2.1.4. 同步 CPU 和 GPU

如 [启动内核](#intro-cpp-launching-kernels) 中所述，内核启动相对于调用它们的主机线程是异步的。这意味着 CPU 线程的控制流将在内核完成之前继续执行，甚至可能在内核启动之前。为了确保内核在主机代码继续之前已完成执行，需要某种同步机制。

同步 GPU 和主机线程的最简单方法是使用 `cudaDeviceSynchronize`，它会阻塞主机线程，直到 GPU 上所有先前发出的工作完成为止。在本章的示例中，这已经足够了，因为 GPU 上只执行单个操作。在较大的应用程序中，可能有多个 [流](./2.5-异步执行.md#252-cuda-流) 在 GPU 上执行工作，`cudaDeviceSynchronize` 将等待所有流中的工作完成。在这些应用程序中，建议使用 [流同步](./2.5-异步执行.md#2524-流同步) API 仅与特定流同步，或使用 [CUDA 事件](./2.5-异步执行.md#253-cuda-事件)。这些将在 [异步执行](./2.5-异步执行.md) 章节中详细介绍。

## 2.1.5. 整合在一起

以下列表显示了本章介绍的简单向量加法内核的完整代码，以及所有主机代码和用于验证答案正确性的实用函数。这些示例默认使用 1024 的向量长度，但接受可执行文件的命令行参数来指定不同的向量长度。


```cuda
#include <cuda_runtime_api.h>
#include <memory.h>
#include <cstdlib>
#include <ctime>
#include <stdio.h>
#include <cuda/cmath>

__global__ void vecAdd(float* A, float* B, float* C, int vectorLength)
{
    int workIndex = threadIdx.x + blockIdx.x*blockDim.x;
    if(workIndex < vectorLength)
    {
        C[workIndex] = A[workIndex] + B[workIndex];
    }
}

void initArray(float* A, int length)
{
     std::srand(std::time({}));
    for(int i=0; i<length; i++)
    {
        A[i] = rand() / (float)RAND_MAX;
    }
}

void serialVecAdd(float* A, float* B, float* C,  int length)
{
    for(int i=0; i<length; i++)
    {
        C[i] = A[i] + B[i];
    }
}

bool vectorApproximatelyEqual(float* A, float* B, int length, float epsilon=0.00001)
{
    for(int i=0; i<length; i++)
    {
        if(fabs(A[i] - B[i]) > epsilon)
        {
            printf("Index %d mismatch: %f != %f", i, A[i], B[i]);
            return false;
        }
    }
    return true;
}

//unified-memory-begin
void unifiedMemExample(int vectorLength)
{
    // Pointers to memory vectors
    float* A = nullptr;
    float* B = nullptr;
    float* C = nullptr;
    float* comparisonResult = (float*)malloc(vectorLength*sizeof(float));

    // Use unified memory to allocate buffers
    cudaMallocManaged(&A, vectorLength*sizeof(float));
    cudaMallocManaged(&B, vectorLength*sizeof(float));
    cudaMallocManaged(&C, vectorLength*sizeof(float));

    // Initialize vectors on the host
    initArray(A, vectorLength);
    initArray(B, vectorLength);

    // Launch the kernel. Unified memory will make sure A, B, and C are
    // accessible to the GPU
    int threads = 256;
    int blocks = cuda::ceil_div(vectorLength, threads);
    vecAdd<<<blocks, threads>>>(A, B, C, vectorLength);
    // Wait for the kernel to complete execution
    cudaDeviceSynchronize();

    // Perform computation serially on CPU for comparison
    serialVecAdd(A, B, comparisonResult, vectorLength);

    // Confirm that CPU and GPU got the same answer

    float* devA = nullptr;

    cudaMallocHost(&C, vectorLength*sizeof(float));

    initArray(B, vectorLength);

    // Allocate memory on the GPU

    cudaFreeHost(C);

在这些示例中，所有线程都执行独立的工作，不需要相互协调或同步。通常，线程需要与其他线程合作和通信以完成其工作。一个块内的线程可以通过 [共享内存]() 共享数据，并同步以协调内存访问。

块级别最基本的同步机制是 `__syncthreads()` 内建变量，它是一个屏障，块中的所有线程都必须在此等待，然后才允许任何线程继续执行。[共享内存]() 给出了一个使用共享内存的例子。

为了实现高效协作，共享内存应该是每个处理器核心附近的低延迟内存（很像 L1 缓存），并且 `__syncthreads()` 应该是轻量级的。`__syncthreads()` 仅同步单个线程块内的线程。

块之间的同步仅在特定情况下受支持。例如，[线程块簇](../chapter-1-CUDA简介/1.2-编程模型.md#12211-thread-block-clusters-线程块簇) 允许一个簇内的块进行同步，而 [协作组 API]() 提供了创建跨块同步域的机制。

通常，当同步保持在线程块内时，可以获得最佳性能。线程块仍然可以使用 [原子内存函数]() 来处理共同的结果，这将在后续章节中介绍。

[第 3.2.4 节]() 介绍了 CUDA 同步原语，它们提供了非常细粒度的控制，以最大化性能和资源利用率。

## 2.1.6. 运行时初始化

CUDA 运行时为系统中的每个设备创建一个 [CUDA 上下文]()。此上下文是此设备的主上下文，并在第一个需要此设备上活动上下文的运行时函数处初始化。该上下文在应用程序的所有主机线程之间共享。作为上下文创建的一部分，设备代码在必要时进行 [即时编译](../chapter-1-CUDA简介/1.3-CUDA平台.md#1343-just-in-time-compilation-即时编译) 并加载到设备内存中。这一切都是透明地发生的。CUDA 运行时创建的主上下文可以从驱动 API 访问以实现互操作性，如 [运行时和驱动 API 之间的互操作性]() 中所述。

从 CUDA 12.0 开始，`cudaInitDevice` 和 `cudaSetDevice` 调用会初始化运行时以及与指定设备关联的主 [上下文]()。如果运行时 API 请求发生在这些调用之前，运行时将隐式使用设备 0 并根据需要自行初始化。这在计时运行时函数调用以及解释对运行时的第一次调用的错误代码时非常重要。在 CUDA 12.0 之前，`cudaSetDevice` 不会初始化运行时。

`cudaDeviceReset` 会销毁当前设备的主上下文。如果在主上下文被销毁后调用 CUDA 运行时 API，将为该设备创建一个新的主上下文。

> **注意**
> CUDA 接口使用在主机程序启动期间初始化并在主机程序终止期间销毁的全局状态。在程序启动或 main 之后终止期间（隐式或显式）使用任何这些接口将导致未定义行为。
> 从 CUDA 12.0 开始，`cudaSetDevice` 在更改主机线程的当前设备后，如果运行时尚未初始化，则会显式初始化运行时。CUDA 的先前版本将新设备上的运行时初始化延迟到 `cudaSetDevice` 之后第一次调用运行时。因此，检查 `cudaSetDevice` 的返回值以查找初始化错误非常重要。
> 参考手册中错误处理和版本管理部分的运行时函数不会初始化运行时。

## 2.1.7. CUDA 中的错误检查

每个 CUDA API 都返回一个枚举类型 `cudaError_t` 的值。在示例代码中，这些错误通常不会被检查。在生产应用程序中，最佳实践是始终检查和管理每个 CUDA API 调用的返回值。当没有错误时，返回的值是 `cudaSuccess`。许多应用程序选择实现一个实用宏，如下所示：

```cuda
#define CUDA_CHECK(expr_to_check) do {            \
    cudaError_t result  = expr_to_check;          \
    if(result != cudaSuccess)                     \
    {                                             \
        fprintf(stderr,                           \
                "CUDA Runtime Error: %s:%i:%d = %s\n", \
                __FILE__,                         \
                __LINE__,                         \
                result,\
                cudaGetErrorString(result));      \
    }                                             \
} while(0)
```

此宏使用 `cudaGetErrorString` API，该 API 返回描述特定 `cudaError_t` 值含义的人类可读字符串。使用上述宏，应用程序将在 `CUDA_CHECK(expression)` 宏中调用 CUDA 运行时 API，如下所示：

```cuda
    CUDA_CHECK(cudaMalloc(&devA, vectorLength*sizeof(float)));
    CUDA_CHECK(cudaMalloc(&devB, vectorLength*sizeof(float)));
    CUDA_CHECK(cudaMalloc(&devC, vectorLength*sizeof(float)));
```

如果这些调用中的任何一个检测到错误，将使用此宏将其打印到 `stderr`。此宏对于较小的项目很常见，但可以适应更大应用程序中的日志记录系统或其他错误处理机制。

> **注意**
> 重要的是要注意，从任何 CUDA API 调用返回的错误状态也可能指示先前发出的异步操作的错误。[异步错误处理](./2.5-异步执行.md#2542-异步错误处理) 部分将更详细地介绍这一点。

### 2.1.7.1. 错误状态

CUDA 运行时为每个主机线程维护一个 `cudaError_t` 状态。该值默认为 `cudaSuccess`，并在发生错误时被覆盖。`cudaGetLastError` 返回当前的错误状态，然后将其重置为 `cudaSuccess`。或者，`cudaPeekAtLastError` 返回错误状态而不重置它。

使用 [三括号表示法](#21221-三括号表示法) 启动的内核不返回 `cudaError_t`。最佳实践是在内核启动后立即检查错误状态，以检测内核启动或内核启动前的 [异步错误](#2172-异步错误) 中的即时错误。在内核启动后立即检查错误状态时，如果值为 `cudaSuccess`，并不表示内核已成功执行甚至已开始执行。它仅验证传递给运行时的内核启动参数和执行配置没有触发任何错误，并且错误状态不是内核启动前的先前或异步错误。

### 2.1.7.2. 异步错误

CUDA 内核启动和许多运行时 API 是异步的。异步 CUDA 运行时 API 将在 [异步执行](./2.5-异步执行.md) 中详细讨论。CUDA 错误状态在发生错误时被设置和覆盖。这意味着在异步操作执行期间发生的错误只有在下次检查错误状态时才会被报告。如上所述，这可能是对 `cudaGetLastError`、`cudaPeekAtLastError` 的调用，或者任何返回 `cudaError_t` 的 CUDA API。

当 CUDA 运行时 API 函数返回错误时，错误状态不会被清除。这意味着来自异步错误（例如内核无效内存访问）的错误代码将由每个 CUDA 运行时 API 返回，直到通过调用 `cudaGetLastError` 清除错误状态。

```cuda
    vecAdd<<<blocks, threads>>>(devA, devB, devC);
    // check error state after kernel launch
    CUDA_CHECK(cudaGetLastError());
    // wait for kernel execution to complete
    // The CUDA_CHECK will report errors that occurred during execution of the kernel
    CUDA_CHECK(cudaDeviceSynchronize());
    
```

> **注意**
> `cudaError_t` 值 `cudaErrorNotReady` 可能由 `cudaStreamQuery` 和 `cudaEventQuery` 返回，不被视为错误，并且不会被 `cudaPeekAtLastError` 或 `cudaGetLastError` 报告。

### 2.1.7.3. `CUDA_LOG_FILE`

识别 CUDA 错误的另一个好方法是使用 `CUDA_LOG_FILE` 环境变量。当设置此环境变量时，CUDA 驱动程序会将遇到的错误消息写入到环境变量中指定的路径的文件中。例如，以下不正确的 CUDA 代码尝试启动一个线程块，其大小大于任何架构支持的最大值。

## 2.1.10. 线程块簇

从计算能力 9.0 开始，CUDA 编程模型包括一个可选的层次级别，称为线程块簇，由线程块组成。类似于线程块中的线程保证在流式多处理器上共同调度，一个簇中的线程块也保证在 GPU 的处理簇 (GPC) 上共同调度。

类似于线程块，簇也被组织成一维、二维或三维的线程块簇网格，如 [图 5]() 所示。

一个簇中的线程块数量可以由用户定义，CUDA 中支持的最大便携式簇大小为 8 个线程块。
请注意，在太小而无法支持 8 个多处理器的 GPU 硬件或 MIG 配置上，最大簇大小将相应减小。这些较小配置以及支持超过 8 个线程块簇大小的较大配置的识别是特定于体系结构的，可以使用 `cudaOccupancyMaxPotentialClusterSize` API 进行查询。

一个簇中的所有线程块都保证在单个 GPU 处理簇 (GPC) 上同时调度执行，并允许簇中的线程块使用 [协作组]() API `cluster.sync()` 执行硬件支持的同步。簇组还提供成员函数，使用 `num_threads()` 和 `num_blocks()` API 分别查询簇组大小（以线程数或块数计）。可以使用 `dim_threads()` 和 `dim_blocks()` API 分别查询线程或块在簇组中的排名。

属于一个簇的线程块可以访问 *分布式共享内存*，即簇中所有线程块的共享内存总和。一个簇中的线程块能够读取、写入和执行对分布式共享内存中任何地址的原子操作。[分布式共享内存]() 给出了在分布式共享内存中执行直方图操作的示例。

> **注意**
> 在使用集群支持启动的内核中，为了兼容性，gridDim 变量仍然表示以线程块数量为单位的网格大小。可以使用 [协作组]() API 找到块在簇中的排名。

### 2.1.10.1. 在三括号表示法中使用簇启动

线程块簇可以在内核中使用编译时内核属性 `__cluster_dims__(X,Y,Z)` 或使用 CUDA 内核启动 API `cudaLaunchKernelEx` 来启用。下面的示例展示了如何使用编译时内核属性启动一个簇。使用内核属性的簇大小在编译时固定，然后可以使用经典的 `<<< , >>>` 启动内核。如果内核使用编译时簇大小，则在启动内核时不能修改簇大小。

```c++

// Compile time cluster size 2 in X-dimension and 1 in Y and Z dimension
__global__ void __cluster_dims__(2, 1, 1) cluster_kernel(float *input, float* output)
{

}

int main()
{
    float *input, *output;
    // Kernel invocation with compile time cluster size
    dim3 threadsPerBlock(16, 16);
    dim3 numBlocks(N / threadsPerBlock.x, N / threadsPerBlock.y);

    // The grid dimension is not affected by cluster launch, and is still enumerated
    // using number of blocks.
    // The grid dimension must be a multiple of cluster size.
    cluster_kernel<<<numBlocks, threadsPerBlock>>>(input, output);
}
